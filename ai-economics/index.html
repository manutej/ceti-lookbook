01ABSTRACT // EVERY-PROMPT-HAS-A-PRICE UPDATED 2026-05-15 14:32 UTC

Tokens are the new SaaS unit. They are billed by the million, priced asymmetrically per direction, discounted ninety percent when cached, marked up twenty-five percent when written into a cache, halved when batched, and multiplied by reasoning effort when a model thinks before it speaks. The bill bends to whoever knows the levers. This terminal walks an engineering-and-ops audience through the May 2026 pricing surface for frontier models, the four levers that govern unit economics, the arbitrage between model tiers, the exact mathematics of prompt caching, the batch-API stack, output-token discipline, infrastructure costs that hide off-invoice, and a real RAG case study that took daily spend from $42.18 to $4.51 without changing a single user-facing answer.

The thesis: most production AI bills are three to ten times what they should be, not because the model is wrong but because the cheapest token is the one you do not buy twice. Every section that follows is a way to not buy a token twice.

Fourth, more subtle: context size and TTFT trade against each other. Gemini 3.1 Pro at 2M context has a 650ms TTFT; Haiku 4.5 at 200K returns its first token in 200ms. The longer the window, the more state the model attends over before streaming. Mix tiers — Haiku for the chat loop, Gemini Pro for the deep retrieval — and you win both ends.

04THE FOUR LEVERS · WHAT BENDS THE CURVE

Lever one: model selection. The cost difference between Opus and Haiku for the same task — assuming Haiku can handle it — is five times on input and five times on output. The cost difference between Sonnet and Haiku is three times on both. Most production traffic does not need the flagship model; it needs the model that can do the task at the latency budget the product requires. Put a Haiku-class classifier in front of Sonnet and you will route seventy percent of traffic to the cheap path for two-tenths of a cent of routing overhead per request. The savings are immediate and they compound.

Lever two: prompt caching. Anthropic charges 1.25× base on the first request that writes a cache entry and 0.10× base on every request that reads it within the TTL — five minutes on the default tier, one hour on the extended tier. Mathematically: at k cache reads per write, average input cost per request is (1.25 + 0.10·k) / (k+1) times base. At k=5, that is 0.29× — a seventy-one-percent input discount. At k=20, it is 0.15×. The math says: cache anything stable that is read more than five times within the TTL window.

Lever three: batch API. A flat fifty-percent discount in exchange for up to twenty-four hours of latency. Stacks with caching, stacks with model selection. For overnight evals on a Haiku-class model with a cached system prompt and batch submission, you are paying five percent of the synchronous Opus rate. That is the ninety-five-percent corner — the cheapest one-million tokens in the matrix is a batched cached Haiku read at approximately $0.05 per million.

Lever four: output token discipline. Output tokens cost five times input on every Anthropic frontier model and roughly the same multiplier elsewhere. The default max_tokens on most SDKs is high — four thousand or eight thousand — and most production prompts ship with the default. They should not. The right discipline is: pick the smallest output budget that still produces complete answers, prefer structured output schemas (JSON mode forces the model to be concise), and prune system instructions that ask the model to "explain its reasoning" or "be thorough" unless you actually want to pay five-times-input for that explanation.

# k → ∞  →  E → 0.10   (asymptotic floor: the read price itself)

# BREAK-EVEN: cache pays off the moment k ≥ 1.
# ONE re-read inside the TTL window already returns you to base.

The break-even mathematics is, in practice, trivially favorable. As long as the request rate against a cacheable prefix exceeds one read per five-minute window, prompt caching is net positive. For any production chat or RAG system with more than a single concurrent user, that condition is met within the first minute of traffic. The interesting question is not whether to cache; it is what to mark cacheable and where to put the cache boundary.

The right cache boundary is the largest stable prefix of every request: system prompt first, then tool definitions, then few-shot examples, then the RAG document, then the dynamic user message. Each cache_control marker creates a cache point. Anthropic allows up to four markers per request, so nested hierarchies work — system stable across all sessions, document stable within one user's reading session.

The wrong cache boundary puts the user message inside the cached block, or puts a per-session UUID in the prefix. Both invalidate the cache every request and you pay 1.25× write without ever reading it back. The most common caching bug is timestamping the system prompt — a "current time is 2026-05-15 14:32:08 UTC" line that silently breaks the cache. Strip the seconds; ideally strip the time entirely; inject the date once per day server-side outside the cached block.

The one-hour TTL tier costs 1.25× extra on cache write (1.5625× base for the lone write) but extends the read window from five minutes to one hour. At more than twelve reads per hour against the cached prefix, the one-hour tier is cheaper than re-writing every five minutes. For long user sessions or sticky RAG queries, one-hour is the right default. For high-traffic APIs, five-minute is correct because traffic refreshes it organically.

Batch is the single highest-ROI migration most production systems leave on the table. The discount is unconditional — fifty percent off, no quota, no minimum volume, no commitment — and the SLA is generous: vendors quote up to twenty-four hours but most batches return within two to four. The architectural pattern is to triage every request at intake into "user-blocking" or "async-tolerable" and route the second bucket through the batch endpoint. Anything that can wait until tomorrow morning costs half as much. For a typical SaaS product the async-tolerable share is twenty to forty percent of total token spend — overnight reports, weekly digests, evals, re-indexing — and moving that share to batch produces a ten-to-twenty-percent reduction in total monthly spend with no user-visible change.

Operational pattern: queue batch-eligible requests with stable correlation IDs, drain the queue into JSONL once per hour or per day, store the batch job ID, reconcile results when the batch completes. The batch endpoint returns a results file plus a per-line error file; treat the error file as a queue of synchronous fallbacks. With a clean reconciliation layer, batch becomes a routing toggle in front of the same SDK call, not a parallel codebase. Two failure modes: the SLA is "up to 24h" not guaranteed — anything with a hard 06:00 deadline needs a 02:00 submission and a 05:00 sync-fallback timer. And per-line failure rate runs 0.1-0.5% higher than sync; build the retry layer.

08§OUTPUT-TOKEN-BUDGETS // THE 5:1 RULE OUTPUT IS THE EXPENSIVE DIRECTION

Output costs five times input on every Anthropic frontier model. The implication is that a single one-thousand-token answer costs the same as five thousand tokens of context. Most production prompts pay this tax without noticing because the default max_tokens on most SDKs is high — Anthropic's Python SDK ships with no default cap and a hard limit of 8192 unless you set it; OpenAI's defaults vary by model. The discipline is three rules:

01Cap max_tokens to the smallest budget that produces complete answers. For a structured extraction task that returns a 200-token JSON object, max_tokens=256 is correct; max_tokens=4096 is wasteful — the model will not exceed 256 tokens but you have given it permission to ramble if it wants to, and on long-tail queries it sometimes does.

02Prefer structured output schemas over freeform answers. JSON mode (Anthropic's tools with tool_choice, OpenAI's response_format: json_schema) forces the model to be concise because the schema constrains the output shape. The difference between "summarize this article" (typically 400-800 tokens of prose) and "extract title, three key points, and one-sentence takeaway as JSON" (typically 120-180 tokens) is a four-times output savings on the same task with no quality loss.

03Prune system instructions about response length. "Explain your reasoning step by step," "be thorough," "consider all perspectives" — every one of these is a request for five-times-input-cost output. If you want chain-of-thought, ask for it inside a <scratchpad> tag and discard it before returning to the user; better yet, use a reasoning model where the thinking tokens are billed separately (and often more cheaply) than the answer tokens.

The 10× rule of thumb: if you cannot predict how many output tokens a feature will produce, assume cost ≈ 10× average input. A RAG feature retrieving 10K tokens and allowed to ramble produces twenty-five-cent answers on Sonnet that no one notices until the monthly bill. Cap output at 600 and the same feature is a four-cent answer.

One more pattern: streaming and aborting. Vendors bill only tokens emitted before the stream closes, so an abort at token 240 of a 4096-token budget bills 240. Combine with a server-side abort timer at the p95 completion time and you get a hard upper bound on per-request cost.

The per-token bill is the easy line. The off-invoice line — vector storage, hosting, egress, observability, prompt-versioning tooling — is where teams who under-budget get surprised in month three. As a rule of thumb, expect infrastructure to run twenty to forty percent of your total AI spend at small scale and fifteen to twenty-five percent at production scale. The dominant non-token costs are typically (a) the vector database, especially if you self-managed an embedding pipeline and ended up on a managed Pinecone tier; (b) egress from the cloud where your inference lives to wherever your application serves users; (c) the observability stack, where Datadog or LangSmith fees can quietly exceed the model bill if you log every prompt and completion verbatim. The mitigations are predictable: sample logs (1-5% in production is enough to debug), self-host pgvector for under one million embeddings, and prefer edge inference for latency-critical paths.

The vector-database line is the most commonly mis-budgeted. A naive Pinecone setup with one million 1536-dimensional vectors on the p1.x1 starter pod costs seventy dollars per month; at ten million vectors you are at five hundred per month before serving a query. The same workload on pgvector inside an existing Postgres RDS instance is free except for storage and a small CPU bump. For under one million vectors, pgvector is almost always the right call. Above ten million, managed is almost always right.

Egress is the silent killer of multi-region deployments. AWS charges nine cents per gigabyte for general egress. A chat product streaming ten kilobytes per response ten thousand times per day moves one hundred megabytes a day — fine. The same product with image inputs and PDF outputs can move two to five gigabytes per active user per day and egress alone exceeds inference. Mitigations: serve from the same region as customers, prefer Cloudflare Workers (no egress on the standard plan), cache static artifacts at the edge.

The two asymmetries that matter most for unit economics.
