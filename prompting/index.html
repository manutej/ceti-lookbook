Why patterns

Prompts are software. Patterns are the libraries.

A model is a function from text to text. The text you put in front of it is not just phrasing — it is the program. And like any program, good prompts are built from a small set of reusable structures that solve recurring problems: how to make reasoning visible, how to constrain output, how to give the model role and stakes.

These six patterns are the design patterns of prompting. Once you can name them, you can compose them. A production prompt is almost never one pattern — it is a role prefix, plus three few‑shot examples, plus a structured output spec, with a chain‑of‑thought instruction tucked in the middle. Same idea as the Gang of Four. Same payoff: vocabulary buys you leverage.

Chain‑of‑Thought 01

Show the reasoning.

Ask the model to think step by step. Hidden reasoning becomes visible — and verifiable.

Prompt · "Let's think step by step."

2 baskets × 4 apples = 8 apples
8 + 3 = 11 apples total
⇒ 11

Anatomy of a production prompt

    
Anthropic‑style XML tags. Role, examples, schema, thinking — in 12 lines.

    
Three patterns composed: role prefix, two few‑shot examples wrapped in XML tags for unambiguous boundaries, and a structured output instruction. The XML tag convention is what Claude is trained on.


    


// system prompt — composes Role + Few-Shot + Structured Output
<role>You are a meticulous invoice auditor.</role>

<examples>
  <ex>input:  "Acme Corp, $1,200, paid"
       output: {"vendor":"Acme Corp","amount":1200,"status":"paid"}</ex>
  <ex>input:  "Globex — 850 USD — DUE"
       output: {"vendor":"Globex","amount":850,"status":"due"}</ex>
</examples>

<task>Extract the invoice into JSON matching the schema above.
Think step by step inside <thinking> tags before answering.</task>

<input>{{ user_input }}</input>
