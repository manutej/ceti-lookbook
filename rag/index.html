RAG · Retrieval-Augmented Generation

Don't fine-tune. Retrieve. Memory for stateless models.

A language model can't know what you didn't train it on. RAG plugs that gap at inference time: index your knowledge, pull the relevant chunks at query time, and hand them to the model alongside the question. The model gets context. You skip the GPU bill.

The five-stage pipeline

Query in. Grounded answer out.

RAG is a runtime pattern, not a model architecture. Every request flows through the same five stages — embed the question, search the index, augment the prompt with what you found, generate the answer. The model never learns your data; it borrows it.

Zoom in · stage 03

Inside the search step: geometry, not keywords.

Every chunk in your knowledge base lives as a point in a high-dimensional space — usually 1536 or 3072 dimensions. Similar meanings cluster together. "Refund window," "return policy," and "money-back" all land in the same neighborhood, even with zero shared words.

At query time we drop the question into the same space and pull its k nearest neighbors by cosine distance. No keyword match required. The model gets the closest meaning, not the closest spelling.

The four levers

Where RAG quality is actually won or lost.

Every production RAG system tunes the same four things. Get these right and the model has good evidence to work with. Get them wrong and even GPT-5 will hallucinate confidently.

01 · Chunking

How you slice the source.

Fixed-token windows are easy. Semantic chunking respects paragraph boundaries. Hierarchical lets you retrieve a small chunk but pass its full parent for context.

Default: 512 tokens · 50-token overlap
02 · Embedding

Turning meaning into math.

A text encoder maps each chunk to a fixed-length vector. The choice of model decides how well "refund" lives near "return" and how far it sits from "refurbish."

Common: text-embedding-3-large · voyage-3 · Cohere v3
03 · Indexing

Where the vectors live.

A vector DB stores millions of embeddings and answers "give me the k closest" in milliseconds. HNSW or IVF-PQ are the algorithms doing the heavy lifting underneath.

Stack: Pinecone · Weaviate · Qdrant · pgvector

04 · Reranking

Second pass for the top-k.

Vector search casts a wide net (say 50 candidates). A cross-encoder reranker reads each one against the query and reorders by true relevance. Cheap, late, decisive.

Boost: ~15-20% nDCG · Cohere Rerank 3

What it looks like in code

Twelve lines. A working RAG chain.

Load the docs, split them, embed and index, build a retriever, pipe it into a prompt and a model. That's the whole pattern — every production RAG system is some elaboration of these twelve lines.

Swap Chroma for Pinecone? One line. Add a reranker between retriever and prompt? One pipe. Move to streaming? Change .invoke() to .stream().

minimal_rag.pyfrom langchain_community.document_loaders import WebBaseLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_chroma import Chroma
from langchain_core.prompts import ChatPromptTemplate

docs   = WebBaseLoader("https://docs.example.com/returns").load()
chunks = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=50).split_documents(docs)
store  = Chroma.from_documents(chunks, OpenAIEmbeddings(model="text-embedding-3-large"))

retriever = store.as_retriever(search_kwargs={"k": 4})
prompt = ChatPromptTemplate.from_template("Use ONLY this context:\n{context}\n\nQ: {question}")
chain = {"context": retriever, "question": lambda x: x} | prompt | ChatOpenAI(model="gpt-4o-mini")

chain.invoke("What's our refund window for damaged shipments?")
