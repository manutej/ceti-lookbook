// readme.md

What LangGraph is. Why it exists.

LangGraph is a low-level orchestration framework for building stateful, multi-step LLM applications. It treats your agent as a directed graph of nodes and edges with explicit, durable state. Nodes do work (call an LLM, hit a tool, transform data). Edges decide what runs next. State is a typed object that threads through every step, gets persisted on every transition, and survives crashes, restarts, and approval pauses that stretch across days.

It exists because LangChain agents shipped, hit production, and broke. The original AgentExecutor (2022–2023) was a single while-loop that hid every interesting decision behind a string-templated prompt. Teams shipping real products hit the same wall: no way to inspect intermediate state, branch on tool output, pause for a human, recover from a crash, or run specialists in parallel. The agent was a black box. Debugging meant print() statements and prayer.

The response, released mid-2024 and matured through 2025–2026, was a rebuild around Pregel-style graph execution — the model Google uses for trillion-edge graph workloads. Define a state schema, register node functions that return updates, wire them with conditional edges. The runtime handles checkpointing, streaming, parallel branches, recursion limits, and human-in-the-loop pauses. Intentionally low-level: there is no magic_agent() helper. You write the graph; the runtime executes it; you see and modify every step.

Today (early 2026) LangGraph runs production agents at Klarna, Replit, Uber, LinkedIn, Elastic, AppFolio, and Komodo Health, among many others. The langgraph package is the de-facto control plane for the LangChain ecosystem; langchain v1.0 exposes create_agent() as a thin wrapper that compiles a LangGraph underneath. This page walks the runtime: ReAct as a graph, the state schema, checkpointing, streaming, HITL, multi-agent patterns, and the gotchas.

> ReAct loop, drawn as a graph.

Each node receives the full state, returns an update, and hands control to the next node. One tick = one LLM call, one tool invocation, one observation.

  1. REASON The LLM reads conversation, decides next step — or whether it's done.
  2. ACT Agent invokes a tool: search, db query, Python function, external API.
  3. OBSERVE Tool output is appended to state. Loop continues, or routes to END.

// the_react_loop.md

Where the loop came from — Yao et al., 2022.

The cycle above is not a LangGraph invention. It is a direct implementation of the ReAct pattern from "ReAct: Synergizing Reasoning and Acting in Language Models" (Yao et al. — arXiv:2210.03629). The paper shows that interleaving free-form reasoning with grounded tool calls beats either alone on HotpotQA, Fever, ALFWorld, and WebShop. LangGraph is the runtime that makes ReAct shippable.

The ReAct paper's central insight is structural. Before ReAct, the two dominant patterns were Chain-of-Thought (Wei et al., 2022 — think out loud, then answer) and Act-only tool-use (SayCan, WebGPT — pick a tool, run it, repeat). CoT hallucinated freely; Act-only wandered without a plan. ReAct's trick was to alternate: write a Thought, emit an Action and Action Input, receive an Observation, repeat. The thoughts give the model space to plan; the actions give it ground truth; the observations close the loop. On HotpotQA the paper reports a +10 EM jump over CoT-only on the same backbone.

> THE ORIGINAL PROMPT TEMPLATE

The prompt is almost embarrassingly simple. Five labeled lines, repeated. The model learns the pattern from a handful of in-context examples and then continues it. Here is the canonical form, lightly cleaned up:

# SYSTEM PROMPT Answer the following question. You have access to the following tools: {tool_descriptions} Use this exact format. Repeat the Thought/Action/Action Input/Observation block as many times as needed. End with Final Answer. Question: {the_user_question} Thought: <reason about what to do next> Action: {tool_name_from_list} Action Input: {json_args_for_the_tool} Observation: <tool output, injected by the runtime> ... (loop) Thought: I now know the final answer. Final Answer: {the_answer}

That is the entire trick. The runtime parses Action: and Action Input:, dispatches the tool, appends the result as Observation:, and feeds the growing transcript back for the next turn. The loop terminates on Final Answer:. Modern function-calling models (GPT-4o+, Claude 3+, Gemini 1.5+) replace the text-parsing dance with structured tool_call objects, but the four logical sub-steps are unchanged.

> THE FOUR SUB-STEPS, IN DETAIL

[01] THOUGHT — The model emits free-form text explaining what it intends to do and why. This is the reasoning step; thoughts are for working memory and never get passed to a tool. In LangGraph this is the LLM call inside the reason node. Ablation studies in the paper show that removing thoughts drops HotpotQA EM from 35.1 to 25.7.

[02] ACTION — The model selects exactly one tool from the allowed list. With function-calling models this is a strongly-typed dispatch: invalid actions come back as Observation: error: tool not found so the model can recover. The action is a categorical choice — the model commits.

[03] ACTION INPUT — The tool arguments, almost always a JSON object. With Pydantic-typed tools the runtime validates input shape before dispatch; type errors become observations and the model retries. This is where most agent bugs live. Strong schemas + validation feedback + a max-retry budget is the production recipe.

[04] OBSERVATION — The tool result, appended verbatim as a ToolMessage. The model now reads the full transcript and generates the next Thought. Token budget matters here: a long-running agent with verbose observations can blow past the context window. LangGraph's trim_messages helper and per-node summarization nodes solve this.

The paper's most-cited contribution is the interleaving — reasoning informs the action, observation informs the next reasoning step. That feedback loop is what LangGraph encodes as the reason → act → observe cycle. Everything else on this page is plumbing on top.

// state_schema.py

State is a typed object. Reducers merge updates.

Every LangGraph application starts with a state schema. It is a TypedDict (Python) or a Zod schema (TypeScript) that declares the keys your graph cares about and, for each key, the reducer function the runtime should use to merge updates from concurrent nodes. Reducers are the single most important concept in LangGraph — they are what makes parallel branches and partial updates safe.

A state schema declares what the graph remembers between nodes. The simplest is MessagesState, a built-in with one key (messages) and add_messages attached. Real apps need more — tool budgets, step counters, scratch buffers, summaries. Declare the keys, pick a reducer for each, and the runtime threads the typed object through every node call.

$ schema.py LISTING 1 · 12L · STATE DEFINITION 
 1from typing import Annotated, TypedDict
 2from langgraph.graph.message import add_messages
 3from operator import add
 4
 5class AgentState(TypedDict):
 6    # messages: append, dedup by id, keep order
 7    messages: Annotated[list, add_messages]
 8    # documents: concatenate lists across parallel nodes
 9    documents: Annotated[list[dict], add]
10    # scalar: last-write-wins (no reducer)
11    iterations: int
12    next_step: str

The key bit is Annotated[list, add_messages]. The runtime reads the reducer from the annotation and calls add_messages(existing, new) to merge concurrent updates. For plain lists use operator.add. For scalars, last-write-wins is the default. Without a reducer, concurrent updates raise InvalidUpdateError.

> THE add_messages REDUCER, EXPANDED

add_messages is not just concatenation. It deduplicates by message id (so checkpoint replays don't double-append), coerces dicts into BaseMessage subclasses, and honors a RemoveMessage sentinel for id-based deletes. Exactly the reducer you want for conversation transcripts.

$ reducer.py LISTING 2 · 11L · ADD_MESSAGES BEHAVIOR 
 1from langchain_core.messages import AIMessage, RemoveMessage
 2from langgraph.graph.message import add_messages
 3
 4existing = [AIMessage(content="hi", id="a1")]
 5incoming = [AIMessage(content="hi (edited)", id="a1")]
 6add_messages(existing, incoming)
 7# => [AIMessage("hi (edited)", id="a1")]  # dedup + replace by id
 8
 9# Delete a message by emitting a RemoveMessage with its id
10add_messages(existing, [RemoveMessage(id="a1")])
11# => []

> CONDITIONAL EDGES = DECISIONS AS DATA

Once you have a state schema, the second piece is the conditional edge: a small pure function that reads state and returns the next node name (or END). The router is where “if the model called a tool, go act; otherwise stop” lives. Pure, so testable in isolation. Compiled, so missing destinations raise at compile time — not at midnight in production.

$ router.py LISTING 3 · 14L · CONDITIONAL EDGE 
 1from langgraph.graph import StateGraph, START, END
 2
 3def route_after_reason(state: AgentState) -> str:
 4    last = state["messages"][-1]
 5    if last.tool_calls:
 6        return "act"          # model asked for a tool
 7    if state["iterations"] >= 10:
 8        return "summarize"    # budget hit, wrap up
 9    return END                # model is done
10
11graph = StateGraph(AgentState)
12graph.add_node("reason", reason)
13graph.add_node("act", act)
14graph.add_node("summarize", summarize)
15graph.add_edge(START, "reason")
16graph.add_conditional_edges("reason", route_after_reason)
17graph.add_edge("act", "reason")
18graph.add_edge("summarize", END)

Three primitives, infinite topology. The compiled graph is a Pregel-style executor: at each step the runtime invokes the active node, merges updates through per-key reducers, persists a checkpoint, and reads the router to pick the next set.

// runtime_primitives

Three primitives that make agents production-grade.

Checkpointing, streaming, and human-in-the-loop are not separate features bolted on top of the graph — they fall out of the runtime almost for free, because the runtime persists state on every transition and exposes the event stream to callers. Below: what each one does, when you need it, and the minimum code to turn it on.

[01]> CHECKPOINTING

Every node transition writes a checkpoint — a full snapshot of state plus the “next node” pointer — to a configured persistence backend. Set the checkpointer argument at compile time and the runtime does the rest: crash recovery, time-travel debugging, fork-and-resume, and human-in-the-loop pauses are all just “load a checkpoint, change something, continue.”

The persistence model is a write-ahead log per thread_id — a thread is one conversation or workflow instance. Each step appends a checkpoint with a unique checkpoint_id. Resume by reusing thread_id; time-travel by passing a specific checkpoint_id; fork by copying a checkpoint into a new thread and running both in parallel.

$ checkpoint.py SQLITESAVER · LOCAL DEV 
 1from langgraph.checkpoint.sqlite import SqliteSaver
 2from langgraph.graph import StateGraph
 3
 4# 1. Open a checkpointer. Use Postgres in prod.
 5memory = SqliteSaver.from_conn_string("./agent.db")
 6
 7# 2. Compile the graph with it.
 8agent = graph.compile(checkpointer=memory)
 9
10# 3. Use a thread_id to scope a conversation/run.
11cfg = {"configurable": {"thread_id": "user-42-session-7"}}
12agent.invoke({"messages": [("user", "hello")]}, cfg)
13
14# Later (minutes? days? after a crash?) — same thread_id resumes.
15agent.invoke({"messages": [("user", "follow-up")]}, cfg)

Three backends ship in the box: MemorySaver for tests, SqliteSaver for local dev and single-node deployments, and PostgresSaver for production. The Postgres version uses SELECT ... FOR UPDATE SKIP LOCKED so multiple workers can pull from the same queue of pending threads without stepping on each other. The wire format is stable; you can inspect checkpoints with agent.get_state(cfg) and walk history with agent.get_state_history(cfg), which returns the full log in reverse-chronological order. That last call is what powers the LangGraph Studio time-travel UI.

Checkpointing overhead is negligible against the LLM call — a few KB written per step against 100ms+ of inference latency. Teams that skip it “for speed” regret it the first time a process dies mid-run. Turn it on. The default is off only because it requires you to choose a backend.

[02]> STREAMING

Three orthogonal stream modes ship with the runtime, and you can subscribe to any combination. stream_mode="values" emits the full state after every node. "updates" emits only the per-node delta. "messages" emits token-by-token LLM output. Pick what your UI needs and let the runtime do the multiplexing.

The streaming API is the single most under-appreciated piece of the runtime. Without it you have two bad choices: block on invoke() and your UI hangs for ten seconds while the agent thinks, or hand-roll a websocket protocol to push intermediate state from the agent to the client. With it, you call agent.stream(input, cfg, stream_mode="...") and you get an iterator of typed events. Token streaming, intermediate tool calls, node transitions, and final state all flow through the same channel.

$ stream.py MULTI-MODE STREAM 
 1# Mode 1: stream node-by-node deltas (good for activity logs)
 2for chunk in agent.stream(payload, cfg, stream_mode="updates"):
 3    for node_name, update in chunk.items():
 4        print(f"[{node_name}] ", update)
 5
 6# Mode 2: stream LLM tokens (good for chat UIs)
 7async for msg, meta in agent.astream(
 8    payload, cfg, stream_mode="messages"
 9):
10    if meta["langgraph_node"] == "reason":
11        print(msg.content, end="", flush=True)
12
13# Mode 3: stream both at once (multi-stream)
14for evt in agent.stream(payload, cfg, stream_mode=["updates", "messages"]):
15    kind, payload = evt
16    handle(kind, payload)

Stream mode composability makes the “agent activity log + token-by-token response” pattern trivial. Hook updates into an activity panel; hook messages into the current chat bubble. Both are live, merged by sequence number on the client. The same stream() call works locally, behind FastAPI, or behind LangGraph Cloud.

Streaming and checkpointing compose. A streamed run still writes checkpoints at every node boundary, so a disconnected client can replay missed events by walking get_state_history(). That is what makes LangGraph viable for long-running workflows behind flaky browser tabs.

[03]> HUMAN-IN-THE-LOOP

Drop a interrupt() call inside any node and the runtime suspends the graph, persists state, and returns control to the caller. Resume by calling Command(resume=...) with the human's answer. The agent picks up exactly where it left off — minutes or days later.

The first-class HITL primitive is interrupt(). You place it inside a node exactly where a human should weigh in — before a destructive tool, after a draft, when confidence drops. The runtime catches the interrupt, snapshots state to the checkpointer, and returns with an __interrupt__ sentinel telling the caller what input is needed. The graph is paused; nothing burns CPU.

$ hitl.py INTERRUPT + RESUME 
 1from langgraph.types import interrupt, Command
 2
 3def approval_node(state: AgentState):
 4    draft = state["messages"][-1].content
 5    # Pause and surface the draft to the caller
 6    decision = interrupt({"draft": draft, "question": "approve?"})
 7    if decision["approved"]:
 8        return {"next_step": "send"}
 9    return {"messages": [("user", decision["feedback"])]}
10
11# Run hits interrupt() -- agent.stream() returns an __interrupt__ event.
12for evt in agent.stream(payload, cfg): handle(evt)
13
14# Human responds via your UI; you resume with Command(resume=...)
15agent.invoke(
16    Command(resume={"approved": True}),
17    cfg,  # same thread_id -- runtime loads checkpoint
18)

Three placement patterns cover almost every real use case. Pre-tool approval — gate the act node on dangerous tool names; human approves, edits args, or rejects. Draft review — pause in a review node after a writer; human accepts, revises, or rewrites. Steering — pause at uncertain decision points (low logprob, retrieval miss, ambiguous routing); human picks the branch.

What makes this work in production vs. just demoware is that the pause survives everything — crashes, deployments, hours of inactivity. The checkpointer holds suspended state; the resume call is idempotent on thread_id. A Slack approval bot, a daily-email queue, a “approve” button in a dashboard — all the same primitive.

// multi_agent_patterns.md

Three shapes for multiple agents.

Once the single-agent ReAct loop is shippable, the next question is “how do I run several agents together?” LangGraph doesn't prescribe one answer. It gives you a graph DSL and you compose. Three patterns cover the overwhelming majority of production deployments: supervisor, hierarchical, and parallel.

[PATTERN 01] // SUPERVISOR

> One router, N specialists

A single supervisor node reads the conversation, picks which specialist runs next, routes to it. Each specialist is its own subgraph with a focused system prompt and a narrow toolset. Control returns to the supervisor, which either delegates again or terminates. Easiest pattern to reason about and debug — routing decisions are visible in state at every step.

Use when: tasks are heterogeneous and a single agent's prompt would balloon

[ USER ] │ ▼ ╭───────────╮ │ SUPERVISOR│ ◄──────┐ ╰─────┬─────╯ │ │ │ ┌──────────┬───┴───┬──────────┤ ▼ ▼ ▼ │ [research] [write] [critic] │ │ │ │ │ └──────────┴───┬───┴──────────┘ │ [ END ]

[PATTERN 02] // HIERARCHICAL

> Teams of teams

A top-level supervisor delegates to team supervisors, each with its own pool of specialists. Useful when the problem decomposes into sub-domains and each team needs to manage its own scratchpad without polluting the parent state. The win is state isolation — the parent only sees the team's final answer. Compose by nesting compiled subgraphs as nodes.

Use when: complexity has natural sub-domains, each with its own toolset

[ TOP SUPERVISOR ] │ ┌───────────┴───────────┐ ▼ ▼ ╭──────────╮ ╭──────────╮ │ RESEARCH │ │ ENGINEER │ │ TEAM │ │ TEAM │ ╰────┬─────╯ ╰────┬─────╯ │ │ ┌─────┼─────┐ ┌─────┼─────┐ ▼ ▼ ▼ ▼ ▼ ▼ srch read sum plan code test │ │ │ │ │ │ └──┬──┴──┬──┘ └──┬──┴──┬──┘ ▼ ▼ ▼ ▼ team summary ──> PARENT <── team summary

[PATTERN 03] // PARALLEL (FAN-OUT/FAN-IN)

> Race or aggregate

A single node fans out to N specialists running concurrently — the runtime dispatches each as an independent task. Each writes back to a shared key with a commutative reducer (usually operator.add on a list); a fan-in node aggregates. Use for ensemble voting, parallel retrieval across heterogeneous sources, or running the same agent with N prompts and picking the best output.

Use when: specialists are independent and you want wall-clock latency < sum(specialists)

// gotchas.txt

Five mistakes that cost you a weekend.

Common LangGraph footguns, in roughly the order you'll hit them on a real project. None of these are bugs in the framework; they're consequences of the runtime's “everything is explicit” design philosophy. Once you see them once, you stop hitting them.

  1. Your reducer must be commutative for parallel paths [CONCURRENCY]

    When two nodes write to the same key in parallel, a non-commutative reducer (last-write-wins on a scalar, for instance) races on task-scheduling order — tests go flaky and the “winner” is undefined. Fan-out/fan-in patterns require commutative, associative reducers. List concatenation works (order within a node is preserved, the union is what you want). For numeric scalars, prefer min/max/sum.

    Annotate parallel-written keys with operator.add or a custom commutative reducer. If you genuinely need last-write semantics, don't fan out into that key.

  2. Forgetting recursion_limit on a self-looping graph [INFINITE LOOPS]

    LangGraph defaults to recursion_limit=25 — the runtime raises GraphRecursionError after 25 super-steps. People hit this when the model gets stuck calling the same tool repeatedly or a router never returns END. The fix is rarely “raise the limit” — it's adding a step counter and routing to a summarize-and-stop node when the budget is hit.

    Add iterations: int to state, increment in reason, route to END at threshold. Keep recursion_limit as a backstop.

  3. State updates returned as objects, not dicts [TYPE ERROR]

    A node must return a dict with keys matching your state schema (or None for no update). Returning a Pydantic model, a tuple, or a single message raises InvalidUpdateError — easy to miss inside a streamed traceback. TypedDict + mypy catches most of these until conditional returns widen the inferred type.

    Always return {"key": value} — even for one key. Wrap messages in {"messages": [msg]}. Type-check node signatures.

  4. Mutating state in place instead of returning updates [SILENT BUG]

    The state arg is a reference to the runtime's working copy. Mutating state["messages"].append(...) and returning None appears to work — until you turn on checkpointing and discover the persisted checkpoint never recorded the mutation, so a resume reverts it. Two parallel nodes mutating in place corrupt each other silently. Always return updates; never mutate the argument.

    Return {"messages": [new_msg]} and let add_messages merge. Treat the state argument as immutable.

  5. Putting interrupt() outside a node function [BAD PLACEMENT]

    interrupt() only works inside a node function called by the runtime. Calling it from a top-level script, a tool, or a callback raises GraphInterrupt at the wrong level and crashes the run. Tools cannot interrupt; they must return. For a human in the middle of a tool, dispatch from a node that previews the call, interrupts, and runs only on approval.

    Move the interrupt() into a dedicated approval node placed before the tool node. Tools stay pure.

  2. > Define state

    Two keys: a message transcript and an iteration counter. messages uses add_messages so parallel branches and human-injected messages merge correctly; the counter is a plain int (only reason writes it). 

    class State(TypedDict):
    messages: Annotated[list, add_messages]
    iterations: int

  3. > Wire the LLM + tools

    Bind tools so the model knows what's callable. .bind_tools() attaches function schemas; the model emits tool_calls instead of free-text actions. Pass a list to scale to dozens of tools. 

    search = TavilySearchResults(max_results=3)
llm = ChatOpenAI(model="gpt-4o").bind_tools([search])

  4. > Write the nodes

    Three functions. reason calls the LLM and bumps the counter. act runs the requested tool and appends the result. approve is the human gate — interrupt() pauses the graph; the resume value drives continue-or-revise. 

    def reason(state):
    return {"messages": [llm.invoke(state["messages"])],
            "iterations": state["iterations"] + 1}

def act(state):
    last = state["messages"][-1]
    results = [search.invoke(c["args"]) for c in last.tool_calls]
    return {"messages": [("tool", str(results))]}

def approve(state):
    draft = state["messages"][-1].content
    decision = interrupt({"draft": draft})
    return {"messages": [("user", decision["feedback"])]} if not decision["ok"] else {}

  5. > Write the router

    Tool calls → act. Budget hit → approve. Otherwise done. Three lines, pure, testable with handcrafted state. 

    def route(state):
    last = state["messages"][-1]
    if getattr(last, "tool_calls", None): return "act"
    if state["iterations"] >= 5: return "approve"
    return END

  6. > Compile with a checkpointer

    Register nodes, draw edges, attach SQLite, compile. Everything before compile() is configuration; everything after is execution. 

    g = StateGraph(State)
g.add_node("reason", reason); g.add_node("act", act); g.add_node("approve", approve)
g.add_edge(START, "reason")
g.add_conditional_edges("reason", route)
g.add_edge("act", "reason"); g.add_edge("approve", END)
memory = SqliteSaver.from_conn_string("./agent.db")
agent = g.compile(checkpointer=memory)

  7. > Stream a run

    Pass a thread_id, stream the updates, print each node's contribution. The stream pauses on interrupt() — the loop ends with an __interrupt__ event surfacing the draft for review. 

    cfg = {"configurable": {"thread_id": "demo-1"}}
payload = {"messages": [("user", "Research SkyPilot vs Ray for LLM training.")],
           "iterations": 0}
for chunk in agent.stream(payload, cfg, stream_mode="updates"):
    print(chunk)

  8. > Resume after human approval

    Render the draft, collect the decision, resume with Command(resume=...) on the same thread_id. The runtime loads the checkpoint, injects the value, and continues from inside approve as if it never paused. 

    agent.invoke(
    Command(resume={"ok": True, "feedback": ""}),
    cfg,
)
# <= run completes; final state is in agent.get_state(cfg)
