SECTION 01.5 · man 1 claude-code

before there were dashboards there were man pages. one screenful of authoritative, terse, terminally honest documentation. here is claude-code rendered the way /usr/bin/* programs have introduced themselves since 1971.

CLAUDE-CODE(1) General Commands Manual CLAUDE-CODE(1)
NAME
claude-code — terminal-resident agentic work-partner for reading, writing, and running real work on a real machine.
SYNOPSIS
claude [prompt] [--model opus-4.7 | sonnet-4.6 | haiku-4] [--allowed-tools Bash,Edit,Read,...] [--mcp server-name] [--plan] [--continue] [--print] [--cwd path] claude mcp add server-name -- command [args...] claude mcp list claude /skills | /commands | /agents | /workflows claude --help
DESCRIPTION
claude-code is anthropic's coding-and-doing agent that runs inside your terminal. unlike a chat interface, it has direct access to the filesystem (read, write, edit), to shell commands (npm, git, pytest, vercel, anything in $PATH), and to the network (curl, fetch, MCP servers). it operates as a long-running ReAct loop: it observes your project, plans, takes a tool action, observes the result, and either continues or asks you a question. the program is interactive by default. you type a request in english (or any language). claude proposes a plan. if you accept, claude executes each step, surfaces diffs before writing files, runs your tests, and reports back. every action is auditable. you can also drive claude non-interactively with the --print flag for shell pipelines, or persistently with --continue to resume a previous session in the current working directory. claude is model-aware. by default it routes to a strong reasoner (opus-4.7) and falls back to faster models (sonnet-4.6, haiku-4) for trivial work. you can pin a model with --model. configuration is hierarchical and lives in .claude/ directories (project) and ~/.claude/ (global). skills, commands, agents, hooks, and MCP servers are all hot-reloaded.
FILES
.claude/CLAUDE.md project rules & context .claude/settings.json hooks, model, permissions .claude/skills/<name>/ reusable skill packages .claude/commands/<name>.md custom slash-commands .claude/agents/<name>.md specialized sub-agents ~/.claude/ global, inherited by all repos
EXIT STATUS
0 the agent completed the requested task to its own satisfaction. 1 the agent stopped because the user denied a tool call. 2 the agent hit a tool error it could not recover from. 130 user pressed Ctrl-C.
SEE ALSO
claude.com/claude-code, docs.claude.com, sdk(7), mcp(7), git(1), make(1), tmux(1)

SECTION 02 · WHAT IT ACTUALLY DOES

Six jobs that used to need an engineer.

Claude Code isn't a chatbot with extra steps. It has hands. It edits real files on your computer, runs real commands, and reads real codebases. Below: the six things business teams actually use it for.

[01] // WRITE & EDIT FILES

Build a working app from a paragraph.

Describe a tool you wish existed — a client report generator, an inventory dashboard, a webhook router — and watch Claude write the files for it across folders. It plans, drafts, tests, fixes its own mistakes, and shows you the diff before saving anything.

[02] // RUN COMMANDS

Press play on the boring stuff.

Installs, deploys, migrations, backups — Claude runs them, reads the output, and tells you what happened in plain English.

[03] // DEBUG

"It broke. Make it work."

Paste an error message. Claude traces the cause, names the file and line, suggests a fix, and offers to apply it.

[04] // DEPLOY

From draft to live URL.

"Put this on Vercel." Claude configures the project, pushes it, and reports the production URL back. Same workflow for Netlify, Cloudflare, or your own server.

[05] // SEARCH

Find a needle in 10,000 files.

"Where do we set the trial length to 14 days?" Claude reads the whole project and points to the exact line — no grep skills required.

[06] // CHAT + DO

A conversation that leaves footprints.

You and Claude talk back and forth like a Slack thread — but every reply can ship as a file, a commit, a deployment, a sent email. Think of regular Claude as a brilliant assistant who can only point. Claude Code can pick things up.

SECTION 03 · WHERE IT LIVES

The terminal is just a place where text goes in and out.

If "the terminal" sounds intimidating, here's the secret: it's just a chat window. You type, something replies, you read the reply, you type back. The format hasn't changed since 1971 — the same year this monitor was designed.

[ email ] Send a request, wait, read a reply. Asynchronous.
[ slack ] Chat back and forth. Synchronous, words only.
[ chatgpt/claude ] Ask a smart friend. They reply with words and maybe a draft.
[ claude-code ] > Same chat — but the friend now has hands, a filesystem, and a deploy button.
The metaphor: if email is talking, Slack is chatting, and ChatGPT is asking — Claude Code is delegating. It has hands, and it uses them.

You can also use Claude Code in VS Code, your browser, or a desktop app if the terminal is too spartan. The same engine, just dressed up.

SECTION 06 · CONFIGURATION

$ tree .claude/

everything claude-code knows about your project lives in a .claude/ directory. it is plain text, version-controlled, diffable. one folder per concern: skills, commands, agents, hooks. nothing magic.

$tree .claude/ .claude/ ├── CLAUDE.md # project rules + context. read on every session. ├── settings.json # hooks, model defaults, permissions, env ├── settings.local.json # machine-only overrides (gitignored) ├── skills/ │ ├── codebase-to-course/SKILL.md │ └── kenya-hara/SKILL.md ├── commands/ │ ├── ship.md # /ship — build, test, deploy, post-mortem │ └── review.md # /review — multi-agent code review ├── agents/ │ ├── ui-reviewer.md # specialist: 6-pillar visual audit │ └── security-triage.md # specialist: AppSec hardening └── hooks/ ├── pre-tool.sh # block dangerous Bash commands └── post-tool.sh # auto-format on every Edit

> a minimal settings.json:

{ "model": "claude-opus-4-7", "permissions": { "allow": ["Read", "Edit", "Bash(npm test:*)", "Bash(git status)"], "deny": ["Bash(rm -rf:*)", "Bash(curl:*)"] }, "hooks": { "PostToolUse": [{ "matcher": "Edit", "command": ".claude/hooks/post-tool.sh" }] } }

skills are reusable, model-discoverable capability bundles (a SKILL.md + helper files). commands are slash-prefixed prompts (/ship, /review) that compose into workflows. agents are scoped sub-claudes — launched in their own context to handle narrow specialist work. hooks fire deterministically around every tool call. they are how you teach claude to obey the rules of your repo without re-explaining them.

SECTION 09 · WORKFLOWS.sh

$ ls workflows/*.sh

five workflows that ceti-trained teams reach for weekly. each is a script you could literally save as a shell function — or run as a one-shot via claude "<prompt>".

>> 1. refactor.sh

#!/usr/bin/env bash # consolidate a sprawling module after a quarter of feature-shipping. $claude "map src/billing/. identify the 3 worst code-smells. propose a refactor plan that keeps every public API stable. execute step by step, running pytest after each step. stop and ask if any test fails."

> why it works: the agent treats the green test-suite as a tripwire. you get the economic upside of a cleanup (smaller diffs, faster onboarding) without the political risk of breaking prod. perfect for end-of-sprint hygiene days.

>> 2. greenfield.sh

#!/usr/bin/env bash # build a small full-stack app from scratch in one focused session. $claude "scaffold a next.js 15 app called 'lead-router'. forms in /app/intake. POST -> postgres via drizzle. admin table at /admin. shadcn ui. tailwind v4. seed 20 fake leads. deploy to vercel. tell me the URL."

> one good prompt · one running URL. greenfield is where claude-code most obviously beats hand-coding: there's no legacy context to load, the agent gets to choose conventions, and "is it deployed?" is a binary success check. typical end-to-end runtime: 12–30 minutes for a non-trivial MVP.

>> 3. debug.sh

#!/usr/bin/env bash # paste an error, get a hypothesis, get a fix, get a confirmation. $claude "production is throwing <paste stack-trace>. find the suspect code paths. form 3 hypotheses ranked by likelihood. write a failing test that reproduces the bug. then fix it. then confirm the test passes."

> the "3 hypotheses" framing matters — it forces the agent to do differential diagnosis instead of jumping to the first plausible cause. the "write a failing test first" step turns the bug into a regression guard for the next deploy.

>> 4. ops.sh

#!/usr/bin/env bash # build the small internal tool nobody had time to build. $claude "we re-onboard a new client every monday and it takes 2 hours. write a CLI 'onboard <client-slug>' that does the 12 manual steps in our HANDOFF.md. dry-run mode by default. real mode behind --apply."

> ops scripts are the highest-leverage thing claude-code unlocks for non-engineers. every team has 4–6 "we should automate this someday" tasks. someday is now a friday afternoon.

>> 5. review.sh

#!/usr/bin/env bash # multi-pass code review before a PR is opened for human review. $claude "/review HEAD~5..HEAD — run three passes in parallel: (1) correctness & bug risk (2) security & secrets handling (3) performance & complexity. consolidate into REVIEW.md."

> pre-PR review catches roughly 60–70% of the comments a human reviewer would have left, which means the human reviewer can focus on architecture and intent instead of nit-picking. teams that adopt this report PR cycle-time dropping by ~40%.
