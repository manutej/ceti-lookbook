Technical Note No. 10/18

Vectors, Embeddings, & Similarity: A Practical Primer

On the geometry of meaning, the metrics that quantify it, and the data structures that retrieve nearest neighbours at scale.

Author

M. Mulaveesala

Affiliation

CETI.AI Research

Issued

May 2026

Abstract

An embedding is a paragraph compressed into a fixed-length vector of real numbers — coordinates in a space where geometric proximity approximates semantic similarity. The substitution of find similar with find near licenses an entire production stack: retrieval-augmented generation, semantic search, deduplication, recommendation, classification by nearest centroid. None of these functions without three components in lockstep — an embedding model that produces well-separated vectors, a distance metric appropriate to the corpus, and an index that returns nearest neighbours in sub-linear time.

This note treats those three components in sequence. We formalise the four distance metrics in common use (cosine, Euclidean, dot product, Mahalanobis); state the complexity claim underlying Hierarchical Navigable Small World (HNSW) indexes and survey contemporary alternatives (IVF, ScaNN, DiskANN, SPANN); analyse scalar, product, and binary quantisation as memory–recall trade-offs; and compare eight production vector databases. Five recurring pitfalls are catalogued. A reference implementation is provided in Listing 1; the bibliography lists nineteen primary sources.

§1

Background: from one-hot to embeddings

For the first four decades of natural-language computing, words were represented as discrete symbols — atoms with no internal structure, distinguished only by their index in a vocabulary. Under this one-hot encoding a vocabulary of size V = 50,000 yields vectors with 49,999 zeros and a single one. Two distinct words are always orthogonal: the vectors for dog and puppy sit at the same angle as dog and parliament. The geometry encodes nothing.

The first break came with distributional semantics: Latent Semantic Analysis [13] in the late 1980s ran an SVD on a term–document matrix, projecting co-occurring terms to nearby coordinates. The mathematics worked; the SVD did not scale.

Definition 1.1

An embedding is a map f : T → ℝd from a token sequence T to a real-valued vector of fixed dimension d, induced by training a neural network such that semantically related inputs are placed at small mutual distance under a chosen metric.

The modern lineage begins with word2vec [4]: Mikolov and colleagues showed in 2013 that a shallow neural network, trained to predict context words from a centre word (the skip-gram objective), produced 300-dimensional vectors with remarkable linear-algebraic structure. The canonical demonstration — king − man + woman ≈ queen — established that geometric operations on embeddings could recover analogical relations the network was never explicitly trained on. GloVe [14], announced a year later, reframed the same problem as a factorisation of the global co-occurrence matrix.

word2vec and GloVe produced static embeddings — one vector per word-form, the same for bank in river bank and investment bank. ELMo, then BERT [15], then the GPT line discarded that assumption: contextual embeddings emit a different vector per token per sentence, so the index no longer collapses sense-distinctions.

The current frontier is instruction-tuned embedding models — OpenAI’s text-embedding-3, Voyage-3, Cohere Embed v4, BGE-M3 — pre-trained with retrieval as a first-class objective via hard-negative mining and contrastive losses. They accept up to 8,192 tokens and emit 1,024–3,072 dimensions. The Massive Text Embedding Benchmark (MTEB) [16] tracks their relative quality across 56 tasks.

In practice d is small — 1,536 for text-embedding-3-small, 3,072 for the large variant, 1,024 for Voyage-3, 768 for Nomic-Embed. The network never sees the word “dog”; it sees a list of d real numbers. Two paragraphs about pets land closer in ℝd than a paragraph about pets and one about cement trucks. Distance is meaning. Once that is granted, find similar becomes find near, and the rest of the note follows.

§2

Distance metrics, formally

Four quantities dominate. We treat each in turn, with formal definition, geometric interpretation, and a note on the failure mode that distinguishes it from the alternatives. Throughout, fix two embeddings a, b ∈ ℝd.

§2.1Cosine similarity

Equation 2.1 — Cosine similarity

The cosine similarity measures the angle θ between two vectors, ignoring their magnitudes. It is the default for normalised text embeddings.

cos(θ) = a · b ‖a‖ ‖b‖ (2.1)

Cosine similarity ranges in [−1, 1] and is invariant to scaling: cos(αa, βb) = cos(a, b) for any positive scalars α, β. Geometrically, it asks do these vectors point in the same direction? while remaining silent on magnitude. Because instruction-tuned text embeddings encode signal-strength in direction, cosine is the natural default.

When it fails. Cosine treats vectors of magnitude 1.0 and 100.0 identically. If magnitude is informative — TF-IDF where document length is a feature, image embeddings where norm encodes salience — cosine throws away signal. A second failure mode arises with very short queries: a three-token query embedded by a model trained on paragraphs sits in a low-density region, and cosine ranking against longer documents grows unstable. Hybrid retrieval (cosine + BM25) is the standard mitigation.

§2.2Euclidean (L2) distance

Equation 2.2 — Euclidean (L2) distance

Straight-line distance in ℝd. Useful when magnitude is informative — image embeddings, geo-spatial vectors, or any space where the constituents are not unit-normalised.

d2(a, b) = √ ∑i=1d (ai − bi)2 (2.2)

L2 is a true metric — it satisfies positivity, symmetry, and the triangle inequality — which matters for theoretical guarantees on certain index structures (e.g., metric trees). Its range is [0, ∞), and unlike cosine it carries an interpretable unit. For raw image embeddings from a CLIP-class model, L2 often produces better separation than cosine because the network deliberately uses magnitude as a confidence channel.

When it matters. Use L2 whenever the encoder is not trained with a unit-norm constraint, or when downstream geometry (clustering, density estimation, anomaly detection) depends on metric-space axioms. Avoid it on normalised text embeddings: identical rankings to cosine at higher floating-point cost.

§2.3Dot product

Equation 2.3 — Dot product (inner product)

Cheap, fast, and equivalent to cosine similarity when both vectors are unit-normalised. The metric most production indexes evaluate on the hot path.

a · b = ∑i=1d ai bi (2.3)

The dot product is the numerator of cosine. On normalised vectors, (2.1) and (2.3) produce identical rankings and the latter avoids two square-root operations per comparison. Modern embedding APIs — OpenAI’s, Cohere’s, Voyage’s — return L2-normalised vectors by default precisely so that downstream indexes can use the dot product without correctness loss.

Equivalence under normalisation. If ‖a‖ = ‖b‖ = 1, then (2.1) reduces algebraically to (2.3), and (2.2) is monotonic in (2.3) via the identity d2(a, b)2 = 2 − 2(a · b). All three metrics therefore induce identical rankings on a unit-normalised corpus. This is why most production indexes configured with “cosine distance” compute a dot product under the hood — the labels diverge from the kernels.

§2.4Mahalanobis distance

Equation 2.4 — Mahalanobis distance

Distance that accounts for the correlation structure of the embedding space. Rarely used in production today, but the right tool for whitened or PCA-decorrelated spaces.

dM(a, b) = √ (a − b)⊤ Σ−1 (a − b) (2.4)

Where Σ is the corpus covariance matrix. Mahalanobis collapses to L2 when Σ is the identity — i.e., when dimensions are uncorrelated and unit-variance. In practice, embedding dimensions are neither orthogonal nor uniformly scaled; naïve L2 over-weights high-variance directions, and Mahalanobis “whitens” the space by dividing through the covariance.

The practical objection is cost: Σ−1 for a 1,536-dim embedding is a 2.36M-entry dense matrix that must be re-estimated as the corpus drifts. The pragmatic alternative is to apply a whitening transformation once at ingest (PCA or ZCA), after which L2 on whitened vectors recovers Mahalanobis on originals. Production deployments that whiten — face-verification, speaker-ID — report 2–5 point recall gains over raw cosine.

The choice between (2.1)–(2.4) is a deployment question (cf. Fig. 1). For unit-normalised text embeddings the first three coincide and (2.3) is preferred for speed. For raw vision embeddings, prefer (2.2). For whitened spaces, prefer (2.4) — or whiten upstream and use (2.2).

A common pitfall: storing un-normalised vectors and querying with cosine distance. The index will not error, but ranking quality degrades silently as magnitudes diverge. Normalise at ingest, not at query.

§3

Indexing algorithms

A single nearest-neighbour comparison is one dot product — microseconds. Ten million is sixteen seconds, intolerable. The indexing problem is the engineering work of pushing query latency back into the millisecond regime by trading exactness for tractability.

§3.1Brute force (flat index)

The base case: compute the metric between the query and every corpus vector, sort, return top k. Cost is O(nd) per query — sub-millisecond at ~10K vectors, several ms at 100K, collapsed by 1M. Flat is nevertheless the right choice below n ≈ 104: exact, zero tuning, zero build. Faiss, ScaNN, and Qdrant all ship a flat index as their reference baseline; switching to an approximate index without first measuring flat latency is premature optimisation.

§3.2IVF — Inverted File Index

The first move beyond flat is to partition the corpus. IVF [9] runs k-means on the corpus to produce n_list centroids — typically n_list = √n — and assigns each vector to its nearest centroid. At query time, the vector is compared to the centroids (cheap), the top n_probe partitions are selected, and a flat scan runs within those partitions only.

Expected query cost is O(n_list · d) for partition selection plus O(n · n_probe / n_list · d) for the within-partition scan. Setting n_list = √n and n_probe = O(1) yields amortised O(√n). Vectors near a partition boundary may have their true nearest neighbour in an unprobed cell — n_probe is the dominant runtime knob. IVF dominated production until ~2020 because it parallelises trivially across centroids and integrates naturally with product quantisation (§4); its recall sensitivity to clustering quality eventually pushed the field toward graph-based indexes.

§3.3HNSW — Hierarchical Navigable Small World

Definition 3.3.1

A Hierarchical Navigable Small World (HNSW) index is a layered proximity graph in which each layer is a probabilistically sparser subgraph of the layer beneath it. Search begins at the top layer with a single entry point and descends greedily, pruning at each layer to a candidate set of size ef.

Exact nearest-neighbour search is O(n) and untenable at production scale. HNSW [1] amortises this cost by exploiting the small-world property of the embedding graph: in well-clustered spaces, any two points are connected by a short path of greedy hops if long-range edges are interspersed with local edges. HNSW makes this explicit by building a hierarchy of graphs, with the topmost layer sparse and long-ranged, the bottom layer dense and local.

Theorem 3.3.2 — Complexity

For a corpus of n vectors with HNSW parameters M (graph degree) and ef (candidate-list size), expected query time is O(log n) and recall @ k exceeds 0.95 for ef ≥ 64 across standard benchmarks [1, 2].

It is this single result that makes vector search a tractable production primitive: it converts retrieval from a linear scan into a logarithmic-time graph traversal, with controllable accuracy via ef.

Two practical consequences follow. First, ef is a runtime parameter, not a build parameter, and may be tuned per-query — a recommendation request might accept lower recall than a legal search. Second, the graph degree M trades index size against recall: larger M increases memory linearly but recall asymptotically. Production deployments typically settle near M ∈ [12, 48], ef_construction ∈ [100, 200], ef_search ∈ [64, 256].

Remark 3.3.3 — Filtered search

HNSW assumes a homogeneous corpus. When queries carry predicates (“documents authored after 2024”), naïve post-filtering can collapse recall by orders of magnitude — the top-k graph hits may all fail the predicate. Production systems implement pre-filtered traversal by attaching predicate bitmaps to graph nodes; see [3] for the integration in Milvus and Qdrant.

§3.4ScaNN, DiskANN, SPANN

Three contemporary variants address regimes where HNSW degrades.

ScaNN [10] (Google, 2020) combines anisotropic vector quantisation with tree-based search. The defining contribution is the loss function: quantisation error is weighted by its impact on the maximum inner product, not by raw L2, improving recall at aggressive compression. ScaNN dominates the billion-scale, in-memory regime where HNSW graph overhead becomes punitive.

DiskANN [11] (Microsoft, 2019) targets the case where the corpus exceeds RAM. It builds the Vamana graph on disk with long-range edges chosen so a typical query traverses 50–100 nodes, each fetched as a single random read. On NVMe SSDs queries resolve in 1–5 ms — slower than RAM-resident HNSW, with 10–100× the addressable corpus per dollar. SPANN [12] is a hybrid: a small in-memory IVF over centroids with posting lists on disk, extending DiskANN’s economics to the 100B-vector regime.

§4

Quantisation

A 1,536-dim fp32 embedding is 6,144 bytes; ten million such vectors is 61 GB — beyond commodity RAM, expensive on managed tiers. Quantisation compresses each component from 32 bits to fewer, accepting controlled loss of distance fidelity for an order-of-magnitude reduction in memory and a comparable speedup.

Scalar quantisation (SQ) replaces each fp32 component with an int8 (or int4) value via a per-dimension affine quantiser fitted to the corpus. 4× memory reduction at 1–2 points of recall loss. SQ is the default in Qdrant and a single flag in Faiss; enable it unless a benchmark proves otherwise.

Product quantisation (PQ) [3] partitions the d-dim vector into m sub-vectors, runs k-means on each (typically k = 256, fitting in one byte), and stores each vector as m centroid IDs. A 1,536-dim vector with m = 96 becomes 96 bytes — 64× reduction. Distance reduces to table lookups against pre-computed centroid distances. PQ loses 5–10 points of recall but is the standard at billion-scale; Faiss IVF-PQ and Milvus GPU PQ both rest on it.

Binary quantisation (BQ) maps each component to a single bit (sign after centering). A 1,536-dim vector becomes 192 bytes; Hamming distance over xor + popcount runs at billions of comparisons per second on one core. Recall loss is sharp (10–20 points), but BQ is well-suited to two-stage retrieval: shortlist with BQ, re-rank with fp32. Matryoshka representation learning [17] is the model-side complement, training embeddings such that truncating to the first 256 dims retains most of the signal.

The choice depends on corpus size and latency budget. Table 2 tabulates trade-offs for a 10M-vector corpus at d = 1,536.

The vector database landscape

Eight systems dominate production deployments in 2026. They occupy distinct points in a multi-dimensional design space — managed-vs-self-hosted, in-process-vs-server, vector-native-vs-extension, dense-only-vs-hybrid, open-source-vs-proprietary — and the right choice depends as much on the operating model of the consuming team as on benchmark numbers.

Scale: maximum corroborated production cluster size as of Q1 2026. DX: developer ergonomics, normalised across SDK quality, docs, and integration breadth. Managed: maturity and breadth of hosted offerings. Hybrid: support for combining vector search with lexical (BM25/full-text) or structured filters.

Three observations guide selection. First, if the team already operates Postgres, pgvector is the right starting point: it inherits the transactional guarantees and operational maturity, and HNSW support since v0.5 closes the performance gap below 50M vectors. Second, for RAG workloads in the 1M–100M range without dedicated infrastructure capacity, Pinecone or Weaviate Cloud remove the index-tuning overhead at predictable cost. Third, at billion-scale or under strict latency budgets, Milvus and Vespa are the only systems with proven multi-tenant production deployments. Chroma and LanceDB remain the right choice for embedded use — prototyping, desktop, edge — where adding a server is disproportionate. Qdrant occupies the middle ground: faster than Weaviate on most benchmarks, better filtering than Pinecone, but self-hosted for the full feature set.

§6

Embedding model selection

The index is downstream of the embedding. A poorly-chosen embedding model places the wrong neighbours adjacent in ℝd, and no amount of HNSW tuning recovers signal that was never encoded. Four models dominate the production landscape in 2026; they are summarised in Table 4.

MTEB [16] aggregates 56 tasks across classification, clustering, retrieval, reranking, STS, summarisation, and bitext mining. The average is a coarse proxy; the per-task breakdown is what matters — a model ranking first on retrieval but tenth on classification is wrong for a sentiment-routing application.

Three practical heuristics. Match the model to the language: monolingual English models underperform silently on multilingual corpora regardless of MTEB average — Cohere v4 and BGE M3 are safer for cross-lingual retrieval. Match the model to chunk length: a 512-token model cannot embed a legal contract paragraph without truncation; Voyage-3’s 32K window is built for this. Match the model to the latency budget: text-embedding-3-large at 3,072 dimensions doubles bytes-on-wire and index memory; if your benchmark shows a sub-point recall difference, the larger model pays a 6× premium for nothing. Finally, embeddings are not portable across model versions — a corpus embedded by ada-002 must be re-embedded if the application migrates, and the cost at 100M passages is non-trivial.

§7

Pitfalls

Five recurring failure modes account for the majority of vector-search incidents we have triaged at client engagements.

  1. Dimension mismatch. Storing a 1,536-dim corpus and querying with a 1,024-dim vector produces a silent error in some clients (the SDK pads or truncates) and a hard error in others. Always pin the embedding model and dimension in a collection-level constraint; never derive dimension from the first record at runtime.
  2. Wrong distance metric. Storing un-normalised vectors and configuring the index for cosine distance does not fail loudly; rankings degrade as the magnitude distribution drifts. Normalise at ingest time, and assert the unit-norm invariant in a CI check on a sample of stored vectors.
  3. Stale embeddings. The corpus is embedded once and incremented thereafter; the embedding model is silently upgraded by the provider, or the team migrates to a new provider. Old vectors and new vectors now occupy incompatible spaces, and ranking is dominated by which embedding generation the document happened to be processed under. Pin the model version, re-embed on migration, and tag every vector with the model fingerprint.
  4. Post-filtering collapse. Retrieving top-100 by vector distance and then filtering by metadata (date range, tenant ID, language) returns zero results when all top-100 hits fail the filter. Use pre-filtered traversal (Qdrant, Milvus, Vespa, pgvector all support this); if the index does not, retrieve over-aggressively (top-1,000) and accept the latency penalty.
  5. Cold-start chunking. Embedding a 50-page PDF as a single passage produces one vector that averages over fifty topics, useful for nothing. Chunk before embedding — recursive splitters with 200–500 token windows and 10–20% overlap are the standard — and embed each chunk independently. The chunking strategy is a hyperparameter as consequential as the model choice.

Reference implementation

The end-to-end retrieval primitive in Python, against OpenAI embeddings and a local Qdrant instance. The shape of the program is invariant across providers; substituting Pinecone, Weaviate, or pgvector changes only the client constructor and the search call.

Listing 1. Minimal embed-index-retrieve pipeline. openai + qdrant-client, Python 3.12.

 1# 1 — turn text into 1,536-dim vectors
 2from openai import OpenAI
 3from qdrant_client import QdrantClient, models
 4
 5oa = OpenAI()
 6qd = QdrantClient("http://localhost:6333")
 7
 8docs = ["a dog wags its tail", "the truck hauls cement", "pasta needs salt"]
 9vecs = [e.embedding for e in oa.embeddings.create(model="text-embedding-3-small", input=docs).data]
10
11# 2 — index under a collection (cosine distance)
12qd.recreate_collection("corpus", vectors_config=models.VectorParams(size=1536, distance="Cosine"))
13qd.upload_points("corpus", [models.PointStruct(id=i, vector=v, payload={"text": d})
14                          for i, (v, d) in enumerate(zip(vecs, docs))])
15
16# 3 — nearest-neighbour query in milliseconds
17q = oa.embeddings.create(model="text-embedding-3-small", input=["my pet is happy"]).data[0].embedding
18hits = qd.search("corpus", query_vector=q, limit=3)
19print([h.payload["text"] for h in hits])  # → ["a dog wags its tail", …]

The cosine distance (2.1) is selected explicitly at line 12. The HNSW index is built implicitly by Qdrant at upload time; defaults of M = 16, ef_construction = 100 yield the latencies reported in Table 5. To enable scalar quantisation (§4) on a 10M-vector corpus, append quantization_config=models.ScalarQuantization(...) to the collection definition at line 12; no other line changes.
