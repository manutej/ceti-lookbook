> SECTION 01 // VISION

PRESS START
TO SEE.

Vision models don’t see images. They see patches. A Vision Transformer (ViT) chops the input picture into a fixed grid of small squares — usually 14×14 or 16×16 pixels — then flattens each square into a vector and treats the whole sequence exactly like a sequence of word tokens. The transformer doesn’t care whether token 47 is the word kettle or a 14×14 pixel patch of a kettle’s spout. It just attends.

This is why the math is so blunt. For Claude Opus 4.7, a 1000×1000 pixel image costs roughly (width × height) / 750 tokens — about 1,334 tokens. A 1024×1024 image at a 32-pixel patch size is exactly 1024 tokens (32×32 grid). Bigger image → more patches → more tokens → more cost. The model sees in a literal mosaic, and you pay per tile.

The original ViT paper (An Image is Worth 16x16 Words, Dosovitskiy 2021) was startling because it took something the CV community had spent a decade tuning — convolutional inductive biases, spatial pooling, multi-scale pyramids — and showed that none of it was load-bearing at scale. Slice the image, project each slice with a single linear layer, add a positional embedding, and feed the result into the same transformer architecture you’d use for text. With enough data, it wins. Every modern vision encoder — CLIP, SigLIP, the proprietary stacks inside GPT-4o, Claude, and Gemini — is a variation on this recipe.

The pixel grid below is a toy 14×14 patch map. Each lit cell is a patch the model has “attended to” while answering. The actual attention weights look very similar — sparse, regional, focused on the parts of the image that carry signal for the current question. The bright cluster in the middle is the foreground object; the dimmer band lower down is the “totals” row of a receipt. Background patches contribute almost nothing.

OCR walkthrough. Imagine a coffee-shop receipt at 1024×1024px. The image becomes 1,024 patch tokens. The text prompt adds another 14 tokens. The model attends across all 1,038 tokens at once — finding that the cluster of patches around row 18-21 (the totals line) co-activates with the text tokens for total, tax, and JSON. It returns structured output in ~200 output tokens. End-to-end cost on Opus 4.7 at $15/M in: about $0.016. Tesseract OCR + a separate LLM cleanup pass would have been cheaper in dollars but slower in latency, brittler on rotated text, and worse at the harder half of the problem: the reasoning step that decides which line is the “total” in the first place.

What native vision changes is the surface area of feasible products. A “scan your receipt” app in 2021 took a six-person ML team eight months — bounding boxes, layout-aware OCR, brittle field extraction rules, and a long tail of edge cases per merchant. The same product in 2026 is a 12-line script and a describe this receipt as JSON prompt. The unlock isn’t accuracy — classical OCR has been near human-parity on clean text for years. It’s that the model also understands what kind of document it’s looking at, what fields you probably want, and how to format the answer.

Section II · Text

Text is the universal interface.

Filed under: tokenisation, BPE, the boring layer that holds everything together. — CETI bureau, 15 May 2026.

Long before a vision encoder learned to chop pictures into patches, language models had already solved the harder problem: how to turn an essentially infinite vocabulary of human language into a finite, attendable sequence. The answer was Byte-Pair Encoding — a greedy compression scheme that starts with raw bytes and iteratively merges the most-frequent adjacent pairs into a single token. The word multimodal becomes two tokens. The word antidisestablishmentarianism becomes six. The word cat becomes one. Every modern model still tokenises text this way.

This matters for multimodal systems because text remains the connective tissue. An image arrives as 1,334 patch tokens. An audio clip arrives as a sequence of mel-spectrogram tokens. A video arrives as ten frames per second of vision tokens interleaved with audio. None of them mean anything to the model in isolation. What gives them a job — extract the total, summarise the call, flag the moment the chart goes up — is always a text instruction. Text is what tells the model what to do with the other senses.

This is also why text is cheap and the rest is expensive. A token of input text on Claude Opus 4.7 costs about $0.000015. The same token slot occupied by a vision patch costs the same in compute — but you needed ~1,300 of them to encode one screenshot, where a paragraph of text would have used ~80. The bandwidth ratio is roughly 16×. So a useful heuristic: think of every image as “100 paragraphs of text-equivalent cost.” If a single sentence would have answered the question, send the sentence.

→ 8 tokens. < $0.0002.

Notice the second prompt isn’t doing anything fancy. It’s just asking a question in words about pictures. The model knows what “most prominently” means because of language pretraining; it knows what the dashboards look like because of vision pretraining; and the cross-modal attention layer is what lets the two pieces of knowledge meet. The text isn’t just glue — it’s the handle we use to grab the model and point it at a question.

There is also a quieter reason text wins. Of the four modalities, only text comes with a built-in formal evaluator — another language model can grade a text answer with high agreement to humans. Vision, audio, and video outputs are far harder to evaluate at scale, which means almost every production system still funnels its final decision through a text representation, even when the input was a sound or a picture. The world is multimodal; the audit trail is text. Plan accordingly.

This is why every serious multimodal API still puts a messages array of {role, content} at the centre of its surface. Vision, audio, video, structured-output blocks all hang off that spine. The spine is text, and it always will be.

> SIGNAL ACQUIRED. 25ms WINDOWS. LATENCY: 320ms.

For most of the past decade, “voice AI” meant three separate models in a chain: STT (speech-to-text, e.g. Whisper) then an LLM then TTS (text-to-speech, e.g. ElevenLabs). End-to-end latency was 2–3 seconds, prosody was discarded at the STT step, and emotional tone was reconstructed from scratch at the TTS step — usually badly. The user experience was a stilted radio drama where the model couldn’t hear that you sounded annoyed.

Real-time native-audio models broke that pipeline. GPT-4o, Gemini Live, and the upcoming Claude voice mode encode raw audio frames directly — the audio enters the same transformer as text, in the same shared embedding space, and the model speaks back by emitting audio tokens that a small vocoder reconstructs into waveform. Round-trip latency drops to about 320ms, well under the ~500ms threshold where humans start to perceive a conversation as “lagging.”

Under the hood the audio is converted to a mel-spectrogram — a 2D image where the x-axis is time, the y-axis is frequency (warped to match human hearing), and brightness is amplitude in dB. The encoder slices this into 25ms windows with a 10ms hop, producing roughly 100 frame-tokens per second of audio. A two-minute voice clip is ~12,000 tokens — comparable to a long article. The trace above is one second of a single sine sweep; in production what enters the model is the spectrogram of that signal, not the waveform itself.

This is why prosody is preserved: the spectrogram captures pitch, energy, voicing, formant structure — everything that distinguishes “sure” from “sure.” A native-audio model can detect sarcasm, hesitation, and emotional state because the signal was never thrown away at an intermediate text bottleneck. The same architectural choice means the model can respond with prosody — matching your energy, lowering its voice when you whisper, slowing down when you sound confused.

The latency budget for natural conversation is brutal. Humans wait ~200ms between turns. Beyond ~500ms it sounds like a delay. The chained pipeline lived in the 2,000–3,000ms range and could never compete. Native models slot in just under the conscious-detection threshold, and the difference is so large that users describe it not as “faster” but as “the AI woke up.”

The practical implication for product builders: voice agents stop being awkward IVR replacements and start being viable for sales calls, therapy intake, language tutoring, accessibility tools, and customer support. The bottleneck moves from latency to interruption-handling — can the model gracefully shut up when the user starts speaking, can it pick up mid-sentence after a pause, can it tell the difference between “keep going” back-channels (“mm-hm”) and an actual interruption. All three frontier providers expose explicit voice-activity-detection (VAD) and barge-in controls in their realtime APIs; this is now the part you have to design, not the part you have to wait for.

SCENE 04 · VIDEO

Video is just vision with a clock attached.

10 FPS · 1M-token context · whole-movie analysis

A video, for a multimodal model, is a sequence of stills sampled at a fixed rate — each still goes through the same vision encoder as a single image, and the resulting token streams are concatenated in time order. There is no special “video transformer.” The temporal reasoning happens because the language model attends across all the frame tokens in the same context window and notices that token 4,500 from frame 47 looks a lot like token 4,500 from frame 12.

The two knobs that matter are sampling rate and context window. Gemini 3 Pro samples up to 10 FPS — fine enough to catch a basketball shot or a UI animation — while Claude Opus 4.7 typically takes keyframes (one per few seconds, plus shot-change detection) to keep token counts down for long narrative content. Both approaches are correct for different problems.

The math: a 1-minute 1080p video at 1 FPS at ~1,300 tokens per frame is ~78,000 tokens — a substantial but workable load. At 10 FPS it’s 780,000, which only fits comfortably in Gemini 3 Pro’s 1M context or its 2M preview tier. This is why “send the whole film” is a Gemini-shaped problem and “extract one moment from a clip” is everyone’s problem.

The unlock is what becomes tractable. With 1M tokens you can paste an entire 2-hour film and ask at what timestamp does the protagonist first lie? — and get a frame-accurate answer with a one-sentence justification. You can drop a 30-minute Zoom recording and ask summarise each speaker’s top three concerns. You can feed a tutorial video and ask generate the chapter timestamps. The bottleneck is no longer the model; it’s figuring out what to ask.

# Gemini 3 Pro · video query · file upload + prompt from google import genai client = genai.Client() video = client.files.upload(file="interview.mp4") # 28 min resp = client.models.generate_content( model="gemini-3-pro", contents=[video, "At what timestamp does the speaker first mention pricing? Return MM:SS and the exact quote."], config={"video_fps": 2}, # 2 FPS = ~336K tokens for 28 min ) # → "12:43 - 'we charge $0.0015 per thousand input tokens'"

For real-time use cases (live security feeds, sports analysis, driving assistance) the chained approach — sample frames, vision-encode each one, batch-call the LLM — still beats native video on cost. But the gap closes every release. The trajectory is clear: video becomes a first-class input the same way images did between 2023 and 2025.

One subtlety worth naming: long-context video raises evaluation problems that the field hasn’t fully solved. “Needle in a haystack” tests for text are well-established — you hide a sentence in 1M tokens and check if the model can retrieve it. The video analogue (hide a single object in one frame of a two-hour film) is much harder to score reliably, because there are many right answers (which timestamp counts? plus or minus how many seconds?) and the failure modes are subtle (it found the right scene but described it inaccurately). Treat published video benchmarks with the same skepticism you bring to MMLU.

Concretely, consider an end-to-end request: “Here’s a screenshot of my dashboard and a 30-second voice memo of my CEO complaining about it. Tell me, in your own voice, what the top fix is.” The model receives three inputs simultaneously:

(1) the screenshot → 1,334 patch tokens through the vision encoder. (2) the audio clip → 3,000 frame tokens through the audio encoder. (3) the text instruction → ~30 BPE tokens. All three land in the same shared embedding space. Every token can attend to every other token through the same self-attention layer. The model can literally notice that the loudness peak at audio token 1,840 (where the CEO emphasises “the chart”) is temporally correlated with text token 14 (“complaining”) which the vision tokens around patch row 8 (a chart in the upper-middle of the dashboard) co-activate with. Cross-modal grounding falls out of the architecture for free.

It then generates an audio response by sampling output tokens that a vocoder reconstructs into a waveform — same forward pass, same context. The total latency for a request like this on GPT-4o realtime is under 1.2 seconds end-to-end. On a chained system (Whisper → GPT-4 vision → ElevenLabs), it would be 4–6 seconds and three API bills.

ARCH · CROSS-MODAL ATTENTION (SCHEMATIC)

The architectural insight here is small and dull and important. Every modality is just a tokeniser away from being text. Once you have a way to convert pictures, sound, and video into vectors of the same dimensionality as text embeddings, the transformer doesn’t need to be rebuilt. The same attention math works. The same scaling laws apply. The same KV-cache optimisations carry over. This is why progress on multimodal capability has been so fast: nobody had to invent a new model class. They just had to invent better tokenisers.

What changes for builders is the product surface. A “multimodal” app isn’t one with four separate features (an OCR feature, a transcription feature, a video-summary feature, a chat feature). It’s one with a single prompt box that accepts any of those inputs interchangeably and treats the user’s instruction as the spine. The right interaction model is a chat input that lets the user drop a screenshot mid-sentence, attach a voice note, paste a video link, and continue typing — with the model reasoning across all of it as one substance.

Three failure modes are worth flagging before you ship. (1) Modality blindness. Models will sometimes “forget” one modality if the others dominate token-count — a 30-second voice memo (3,000 tokens) plus a one-line instruction can be drowned by a single 4K screenshot (5,000+ tokens). If accuracy on the audio matters, downsample the image. (2) Grounding hallucination. Cross-modal attention is powerful but not infallible: a model can confidently describe a chart label that isn’t there because the text instruction primed it to expect one. Mitigation: ask for verbatim quotes and bounding-box-like locators when accuracy is load-bearing. (3) Eval debt. You cannot evaluate a multimodal pipeline with a text-only eval suite. Build a small (50-200 case) golden set per modality combination you care about, and re-run it on every model upgrade. The cost of skipping this is a silent regression that the user finds first.

This is why down-sampling to 1 FPS or extracting keyframes is the single biggest cost-control lever in multimodal pipelines — and why Gemini’s 1M context window is less about being clever and more about being the only context window large enough to hold a real video at useful frame rates.

06 · THE FRONTIER, 2026

Four frontier multimodal models.

No single model wins every modality. Most production stacks now route per-modality: Gemini for video, GPT-4o for real-time audio, Claude for OCR and structured vision, Llama for self-hosted vision. 07 · WHEN TO USE WHAT

Pick by problem shape, not vendor loyalty.

A useful heuristic: if the answer fits in one sentence and the input is a single screenshot, almost any frontier model works and you should pick on price. If the input is a 30-minute recording or a feature-length video, only Gemini’s context window currently keeps the prompt one-shot. If the user-perceived latency budget is sub-second and the channel is voice, GPT-4o realtime is the only production-ready option as of this writing. If the data must never leave your network, Llama 3.2 Vision is the only frontier-grade open option — and the only one where you own the model weights long after the vendor moves on.
