The | operator isn't a string-template hack. It's Python's __or__ overloaded so every Runnable composes with every other Runnable — and the result is itself a Runnable. Sequences compose into sequences compose into pipelines.

That single property is why LCEL chains support streaming, batching, async, and tracing for free. You write one definition; LangChain gives you four execution modes without a second line of code.

The composition primitive

Why __or__ was the key abstraction.

Most frameworks make you learn a config DSL. LCEL gives you one Python protocol — Runnable — and one operator. Everything else is consequences.

Before LCEL, a LangChain pipeline was a graph object. You constructed it with named arguments, registered callbacks, and prayed that the order matched the abstract structure in the docs. The DSL was opinionated, the type system invisible, and refactoring meant rewriting the whole graph.

The Runnable interface changed that. A Runnable is any object exposing four methods — .invoke, .batch, .stream, .ainvoke — plus their async cousins. That contract is the only thing the framework requires. Everything in LangChain ships as a Runnable: prompts, models, retrievers, parsers, tools, even arbitrary Python functions wrapped in RunnableLambda.

The pipe operator is just Python operator overloading. Runnable.__or__ returns a new RunnableSequence whose input is the left operand's input and whose output is the right operand's output. Because the result is itself a Runnable, you can keep piping. The whole pipeline composes algebraically, the way functions compose in mathematics — and that algebra is what lets the framework derive streaming, batching, async, and tracing without any extra code from you.

This matters because composition is what makes systems refactorable. When every step is a Runnable, you can move a piece, swap a piece, wrap a piece in a retry, or run two pieces in parallel — and the rest of the pipeline keeps working. The contract is small enough to memorize and strict enough that it actually composes. That is the entire secret.

runnable_protocol.py
from typing import Protocol, AsyncIterator, Iterator, Any

class Runnable(Protocol):
    # synchronous
    def invoke(self, input: Any, config: dict = None) -> Any: ...
    def batch(self, inputs: list, config: dict = None) -> list: ...
    def stream(self, input: Any, config: dict = None) -> Iterator[Any]: ...

    # async
    async def ainvoke(self, input: Any, config: dict = None) -> Any: ...
    async def abatch(self, inputs: list, config: dict = None) -> list: ...
    async def astream(self, input: Any, config: dict = None) -> AsyncIterator[Any]: ...

    # the operator that makes it composable
    def __or__(self, other: "Runnable") -> "RunnableSequence": ...

The four execution modes — in depth

Same chain. Four shapes of latency.

The verb you choose determines whether your user waits 4 seconds in silence, sees the first token in 200ms, or fans out a hundred queries in the time of one. Pick the wrong one and the pipeline is still correct — just slow, or rude.

chain.invoke(input)

Synchronous call. Blocks until done.

The simplest verb. Pass an input, the pipeline runs through every stage in order, you get the final output. Use it in scripts, notebooks, batch jobs, and any context where the caller cannot proceed without the full answer. Latency is the sum of every stage's latency — typically 2–8 seconds end-to-end for a RAG chain with one LLM hop.

→ p50 latency: 2000–8000ms (LLM-dominated) 
# single answer, single thread
answer = chain.invoke(
    {"question": "refund policy?"}
)
print(answer)
chain.batch(inputs)

Many inputs, concurrent execution.

Pass a list, get a list back — but the framework runs them in parallel using a thread pool (or the underlying client's connection pool). Wall-clock time approaches the slowest single call rather than the sum of all calls. Use it for evals, bulk classification, daily report generation, or any time you have N independent inputs and one pipeline. The max_concurrency config caps simultaneous in-flight requests so you don't get rate-limited.

→ 100 inputs in ~1.2× the latency of one 
# 100 questions, 1× the wall time
results = chain.batch(
    questions,
    config={"max_concurrency": 10}
)
chain.stream(input)

Token-by-token. First byte fast.

Returns an iterator that yields chunks as the model produces them. Time-to-first-token is typically 200–400ms instead of multi-second silence. Every UI that mimics ChatGPT depends on this. Streaming works because the Runnable protocol surfaces partial outputs upstream — a parser at the end can stream partial JSON, a model in the middle streams tokens, and the pipeline coordinates the flow without you writing a single callback.

→ first chunk: 200–400ms; total: same as invoke 
# yield chunks as they arrive
for chunk in chain.stream(question):
    print(chunk, end="", flush=True)
chain.ainvoke(input)

Async / non-blocking. For web servers and agents.

The async cousin of .invoke. Drop it into any async def handler — FastAPI, Starlette, an agent loop, a websocket worker — and the event loop stays free to do other work while the LLM call is in flight. Every other verb has an async twin: abatch, astream, astream_events. Use async whenever you cannot afford to block a server thread, which is essentially anywhere outside a CLI.

→ same latency, frees the event loop 
# non-blocking in async contexts
async def handler(req):
    out = await chain.ainvoke(req.body)
    return JSONResponse(out)

LCEL building blocks

Four primitives shape every chain.

Beyond the basic pipe, LCEL ships a small vocabulary of Runnable wrappers that handle the three things a real pipeline always needs: parallel fan-out, pass-through plumbing, plain Python functions, and conditional routing.

A linear a | b | c chain gets you the demo, but production pipelines have branches: retrieve docs and get user history at the same time, send the same draft through two models for comparison, route easy questions to a cheap model and hard ones to an expensive one. The building blocks below handle every one of those cases — and because each block is itself a Runnable, they compose with each other and with the pipe operator. There is nothing else to learn.

RunnableParallel

Fan out the same input to multiple branches and collect the results into a dict. Use it whenever a downstream stage needs two upstream outputs — the canonical case is RAG, where the prompt template needs both the retrieved context and the original question.

parallel = RunnableParallel(
    context=retriever,
    question=RunnablePassthrough(),
)
# input → {context, question}
"refund?"
⇒
{ context: [...] }
{ question: "..." }
RunnableLambda

Wrap any plain Python function so it implements the Runnable protocol. The cheapest escape hatch in the framework — drop in a one-line transform, a normalizer, a logger — without writing a class or a custom Runnable subclass.

normalize = RunnableLambda(
    lambda q: q.strip().lower()
)
chain = normalize | prompt | model
" REFUND? "
→
"refund?"
RunnablePassthrough

The identity function. It returns whatever it receives. Useful inside a RunnableParallel when you want to forward the original input to a later stage alongside transformed copies. The .assign() variant adds new keys to a dict without removing the originals.

chain = RunnablePassthrough.assign(
    summary=summarizer
) | next_stage
# {x, y} → {x, y, summary}
{ x, y }
→
{ x, y, summary }
RunnableBranch

Conditional routing. Each branch is a (predicate, Runnable) pair, and the first matching predicate wins. Use it to route by intent, language, complexity, or any other dispatch you would otherwise express with a long if/elif tree — without breaking composition.

router = RunnableBranch(
    (is_code, code_chain),
    (is_math, math_chain),
    default_chain,  # fallback
)

What it looks like

Ten lines. A real RAG chain.

Take a question, pull related docs, format both into a prompt, send to the model, parse the result. Every | is a hand-off. Swap ChatOpenAI for ChatAnthropic? One line. Add a re-ranker? One pipe.

rag_chain.py
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_openai import ChatOpenAI

prompt = ChatPromptTemplate.from_template(
    "Answer using only this context:\n{context}\n\nQ: {question}"
)
model  = ChatOpenAI(model="gpt-4o-mini")
parser = StrOutputParser()

# the whole pipeline — five components, four pipes
chain = {"context": retriever, "question": passthrough} | prompt | model | parser

chain.invoke("What's our return policy?")

A real RAG pipeline — annotated

Fifteen lines, six stages, one data flow.

The toy RAG chain above is the shape of the answer, but production pipelines have one or two more stages — a query rewriter at the front, a re-ranker after retrieval, a JSON parser at the end. Here is the same pattern, expanded.

Each piece below is independently testable. You can call retriever.invoke("...") on its own to inspect what came back. You can call rewriter.invoke("...") to see how the query got rephrased. Every stage of a real LCEL chain is a Runnable in its own right, so you debug by isolating, not by adding print statements. The annotated panel on the right walks the input through every transformation.

This is the productivity gain that justifies the abstraction. Once your team writes pipelines this way, swapping a vector store, adding a re-ranker, or fanning out two parallel queries is one line. The interface is small, the operations are explicit, and the data flow is visible at the top of the file rather than scattered across helper functions.

production_rag.py
from langchain_core.runnables import RunnablePassthrough, RunnableParallel
from langchain_core.output_parsers import JsonOutputParser
from langchain_openai import ChatOpenAI

rewriter = rewrite_prompt | ChatOpenAI(model="gpt-4o-mini") | StrOutputParser()
retriever = vectorstore.as_retriever(search_kwargs={"k": 12})
reranker  = CohereRerank(top_n=4)

rag_chain = (
    RunnableParallel(
        rewritten=rewriter,
        original=RunnablePassthrough(),
    )
    | RunnablePassthrough.assign(
        docs=lambda x: reranker.compress_documents(
            retriever.invoke(x["rewritten"]),
            query=x["original"],
        )
    )
    | answer_prompt
    | ChatOpenAI(model="gpt-4o", temperature=0)
    | JsonOutputParser()
)

rag_chain.invoke({"question": "refund policy?"})

Data flow per stage

  1. Rewrite the user's terse question into a fuller search query.
  2. Fan out via RunnableParallel — keep both rewritten and original around.
  3. Retrieve 12 candidates from the vector store using the rewritten query.
  4. Re-rank candidates against the original question; keep top 4.
  5. Format retrieved docs + original question into the answer prompt.
  6. Generate the final answer with a stronger model and temperature 0.
  7. Parse the model's JSON output into a typed dict ready to return.

Streaming patterns

First byte fast. Intermediate events visible.

Streaming isn't just about the final tokens. The Runnable protocol exposes a richer event stream that lets a UI show progress through every stage of the pipeline — query rewriting, retrieval, generation — not just the model output.

The simple case is chain.stream(x), which yields output chunks as soon as the model produces them. In a Server-Sent Events handler, you pipe those chunks straight into text/event-stream and a browser shows tokens appearing one by one. Time-to-first-token drops from "the user is staring at a spinner for four seconds" to "the user sees the first word in 250ms." That single change reshapes how the product feels.

The richer case is chain.astream_events(x, version="v2"). This yields a stream of typed events covering every stage of the chain: on_retriever_start, on_retriever_end, on_chat_model_stream, on_chat_model_end, and so on. You can use this to show "Searching docs..." then "Found 4 sources" then start streaming the answer — all in a single async iterator. The pattern below is what production chat UIs ship.

For structured output, partial JSON streaming is the third pattern. JsonOutputParser emits partial dicts as the model produces them, so you can render a half-built object — fields filling in left to right — while the model is still generating. The combination of token streaming, intermediate events, and partial JSON is what makes modern LLM UIs feel responsive instead of laggy.

streaming_handler.py
from fastapi.responses import StreamingResponse

async def stream_chat(question: str):
    async def event_source():
        async for event in chain.astream_events(question, version="v2"):
            kind = event["event"]

            if kind == "on_retriever_end":
                docs = event["data"]["output"]
                yield sse("sources", [d.metadata for d in docs])

            elif kind == "on_chat_model_stream":
                chunk = event["data"]["chunk"].content
                yield sse("token", chunk)

            elif kind == "on_chain_end":
                yield sse("done", {})

    return StreamingResponse(event_source(), media_type="text/event-stream")

LCEL vs legacy chains

Five dimensions where the abstraction earns its keep.

Old-style LangChain (the Chain base class, LLMChain, SequentialChain) still works — but every dimension that matters for shipping is where LCEL pulls ahead.

The legacy chain classes were a graph-building DSL: you instantiated a chain object, passed it sub-chains as keyword arguments, and the framework ran them in a fixed order. It worked for the demos. It did not survive contact with streaming UIs, async web servers, or production observability needs. LCEL replaces that DSL with a single algebra — and the table below is what that algebra buys you.

Attribute Legacy Chain LCEL Runnable
Composability Hard-coded sub-chain wiring Algebraic — any Runnable pipes into any other
Type safety String-keyed dicts at every hop Typed I/O via Pydantic schemas + IDE inference
Debugging Verbose logs, no per-step trace LangSmith auto-traces every Runnable
Streaming Manual callback handlers .stream() and .astream_events() built in
Deployment Wrap the chain in your own FastAPI handler LangServe exposes any Runnable as a REST endpoint

Drop-in swaps

Change a component. Keep the pipeline.

Every stage of an LCEL chain is interchangeable as long as its inputs and outputs line up. Move from one provider to another, or one storage backend to another, with a single import change.

Stage Was Becomes Diff
Model ChatOpenAI ChatAnthropic one import, one constructor
Retriever Chroma.as_retriever() Pinecone.as_retriever() one binding, same interface
Parser StrOutputParser JsonOutputParser shape of .invoke() changes, rest is identical
Execution chain.invoke(x) chain.stream(x) verb only — return type adjusts

Integration ecosystem

One contract, every provider.

LangChain split its monolith into partner packages — each provider lives in its own pip-installable module that implements the Runnable contract. You depend on what you use; nothing more.

The architectural choice that made this scale: langchain-core holds the Runnable protocol and the base interfaces; langchain holds the high-level chains; and every model provider ships its own thin package — langchain-openai, langchain-anthropic, langchain-google-vertexai, langchain-aws, langchain-cohere, langchain-mistralai, and dozens more. Each package owns its own release cadence and dependency tree, so a security patch in the AWS SDK does not force a re-test of the entire framework.

The same split applies to vector stores (langchain-chroma, langchain-pinecone, langchain-qdrant), document loaders, retrievers, and tools. The ecosystem is large — over 700 integrations as of 2026 — but you only install the leaves you need. A typical production RAG service pulls in three or four partner packages, not the whole tree.

Because every partner package implements the Runnable contract, swapping providers is genuinely one line. Migrate from OpenAI to Anthropic? Change from langchain_openai import ChatOpenAI to from langchain_anthropic import ChatAnthropic and update the constructor. The chain itself never moves. That is the practical payoff of having a stable interface that every integration agrees to honor.

Things people get wrong

What LCEL is, and isn't.

It isn't magic — it's operator overloading.

The pipe character is Python's __or__ dunder. LangChain implements it on the Runnable base class. Nothing more, nothing less. If you understand operator overloading, you understand LCEL.

The four execution verbs are free.

You don't write streaming logic. You don't write batching logic. You define the chain once; LangChain derives .invoke, .batch, .stream, and .ainvoke from the composition.

Types matter at the seams.

A chain is only as composable as its boundaries. If stage N returns a dict and stage N+1 expects a string, you'll see a runtime error. LangChain's type hints help — pay attention to them.

Tracing is the killer feature.

Because every step is a Runnable, LangSmith can trace every hop without instrumentation. You see prompts, retrievals, tokens, latency — per node, per run. This alone justifies the abstraction.

Common pitfalls

Five mistakes the framework will not protect you from.

LCEL is elegant when it works and confusing when it doesn't. The errors below account for most of the support questions on the LangChain Discord — fix them once and they stop happening.

  1. 01

    Mixing sync and async runnables in the same chain.

    If you pipe a sync Runnable into an async one and then call .ainvoke(), the sync stage runs in a worker thread — which is correct, but blocks one worker per call. Latency under load suffers in a way that does not show up in single-shot benchmarks.fix: make every stage async when you intend to call ainvoke; use RunnableLambda(func).with_config(run_async=True) if you must wrap a sync function.

  2. 02

    Forgetting to type your state with Pydantic.

    Untyped dict-passing between stages works until a refactor changes one key name and three downstream stages silently start returning empty strings. LangSmith traces will show garbled outputs but the chain never raises.fix: declare a Pydantic BaseModel for the chain's input and output and use .with_types(input_type=..., output_type=...).

  3. 03

    Forgetting max_concurrency on .batch().

    Default concurrency is unbounded. Batch 500 inputs against a provider with a 60 RPS limit and you will get a wall of 429 errors and a partial result list with retries baked in unpredictably.fix: always pass config={"max_concurrency": N} where N is below your provider's per-second cap.

  4. 04

    Streaming with a parser that buffers.

    Some parsers — most commonly older custom ones, or PydanticOutputParser when given a strict schema — wait for the full output before returning. You think you're streaming; users see no chunks until the entire response is done.fix: use streaming-aware parsers (StrOutputParser, JsonOutputParser) and verify with astream_events that on_chat_model_stream events surface.

  5. 05

    Calling external APIs inside a RunnableLambda without retries.

    Wrapping a third-party call in RunnableLambda is the easiest escape hatch — and it inherits zero of the production hardening you'd get from a proper integration package. Network blips become 500s.fix: wrap with .with_retry(retry_if_exception_type=(httpx.HTTPError,), wait_exponential_jitter=True, stop_after_attempt=3).

Production checklist

Eight things to verify before you ship.

A working LCEL chain in a notebook is not a production system. The checklist below is the minimum bar before exposing a pipeline to real users.

Every item maps to a class of incident we've seen in production: rate-limit cascades, untraceable hallucinations, silent retrieval misses, cost overruns from looping agents, and chains that work fine on developer machines and fall over the moment they hit real traffic. Treat the list as gating, not aspirational.

  • Retries with backoff on every external call (.with_retry on each Runnable that hits a provider).
  • Timeouts per stage via .with_config(run_timeout=...) — never trust a provider to fail fast.
  • LangSmith tracing enabled in production with LANGCHAIN_TRACING_V2; sample if volume demands it.
  • Prompt and embedding caching turned on (set_llm_cache, vendor-side prompt caching for Claude / OpenAI).
  • Concurrency caps on every .batch() call and on the agent loop to avoid rate-limit cascades.
  • Output schema validation with Pydantic so malformed model outputs raise instead of silently corrupting downstream state.
  • Cost ceilings per request (max tokens, max tool calls, max agent steps) — guard against runaway loops.
  • Eval suite in CI — a small, fast LangSmith dataset that runs on every PR so regressions show up before merge, not in production.
