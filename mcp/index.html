On the Model Context Protocol · Essay · No. 03

USB-C for LLMs.

An open standard, and what it changes.

By Manu Mulaveesala  ·  CETI.AI  ·  Spring 2026  ·  22 min read

The day Anthropic released the Model Context Protocol, in November of 2024, nothing felt particularly seismic. There was a spec, a handful of reference servers, four SDKs, and a short blog post. The press did not bite. Most of the people who would later come to depend on MCP did not yet know they were waiting for it. And yet, looking back from 2026, that was the day the integration problem quietly stopped being interesting — which, in software, is often the highest praise an idea can earn.

For two years before that release, every team building with large language models had been writing the same kind of glue code. Each new tool the model needed to touch — a calendar, a database, a ticket system, a file on disk — required bespoke wiring: an API client, an authentication dance, a function description hand-written into the prompt, and a parser to coax the model's JSON back into something the rest of the program could use. Multiply by every model, every framework, every vendor. The number of integrations grew as N × M, where N was the count of applications and M was the count of LLMs. It did not scale, and everyone knew it did not scale, and everyone kept building anyway because there was no shared floor to stand on.

MCP is the shared floor. The conceit is borrowed openly from hardware: before USB-C, every device shipped with its own bespoke cable, and the industry tolerated the absurdity for decades. After USB-C, the cable became infrastructure — invisible, assumed, uninteresting. MCP proposes the same compression for software agents. One protocol, spoken on both ends, collapsing N × M integrations into N + M. A model learns to speak MCP once. A service exposes itself over MCP once. The combinatorics dissolve.

The architecture is deliberately small. There are clients — programs like Claude Desktop, Cursor, or any custom agent — and there are servers, which wrap a tool, a data source, or a workflow. They communicate in plain JSON-RPC 2.0, over one of two transports: stdio when the server runs locally as a spawned process, or HTTP+SSE when it lives behind a network. The model on the client side does not need to know which transport is in use. It does not need to know what language the server is written in. It only needs to know how to ask.

What it asks for is one of three things, and the entire specification turns on these three primitives. Tools are things the model can do: side-effecting functions like create_issue or send_email, each declared with a JSON-Schema so the model can call it correctly on the first try. Resources are things the model can read: a file at file://repo/README.md, a row in a database, a fresh response from an API — read-only context the model can pull in without ever being granted write access. Prompts are workflows the server provides: reusable templates like /git-summarize or /code-review that the user can invoke as a starting point. That is the whole spec. The discipline of refusing a fourth primitive is, I think, why it caught on where earlier tool-calling proposals stalled.

I · AntecedentsA short history of LLM tool use.

To understand why MCP feels inevitable in hindsight, it helps to remember how recently the idea of an LLM doing anything at all was novel. As late as the spring of 2023, a language model was, in the public imagination, a very fluent autocomplete. It could write a sonnet about your dog, or summarize a contract, or hallucinate a citation, but it could not check the weather. It had no hands. The frontier model providers had not yet given it any, and the rest of us were busy trying to figure out whether prompts alone could substitute for proper engineering — a question the answer to which was, predictably and disappointingly, "almost, but not quite."

OpenAI shipped function-calling in June of that year, and it changed the conversation almost overnight. The mechanism was simple: alongside the user's prompt, a developer could pass a list of function signatures — names, parameter schemas, short descriptions — and the model would, when it judged the moment right, emit a structured JSON object naming a function and its arguments. The runtime would execute that function, hand the result back into the next turn, and the model would continue as if nothing unusual had happened. For the first time, the LLM had hands. Or, more accurately, it had the ability to ask politely for a hand and trust that the runtime would lend it one.

What followed was, in retrospect, a Cambrian explosion of incompatible plumbing. LangChain wrapped functions in Tool classes, each with its own conventions for description, retry, and error handling. LlamaIndex did its own thing for retrieval. Every major framework — Haystack, Semantic Kernel, Autogen, CrewAI, Letta — invented a slightly different abstraction over the same underlying capability. OpenAI then introduced Plugins, which lived briefly inside ChatGPT and required a manifest at /.well-known/ai-plugin.json and an OpenAPI spec and a special-purpose hosting arrangement, and then Plugins were quietly retired in favor of GPTs, which were also a kind of tool-use system but with different conventions, and so on. Anthropic shipped tool use of its own in mid-2024 with a cleaner schema but the same fundamental shape. Each system worked. None of them worked with each other.

By the autumn of 2024, the cost of this fragmentation had become impossible to ignore. A team that wanted Claude, GPT-4, and an open-source model to all be able to read from the same internal wiki had to write three separate adapters, maintain three sets of credentials, and reconcile three slightly different ideas of what a "tool result" looked like in the conversation. Worse, the tools themselves were not reusable across teams: each company's GitHub integration was a snowflake, each customer's Salesforce wrapper was hand-rolled. The industry had collectively built the same set of integrations several thousand times, often badly, and the cost of not having a standard was being paid mostly by the small companies who could least afford it. The large labs could absorb the duplicated engineering; an early-stage team trying to ship an agent could not. What everyone wanted, without quite saying so, was a way to stop solving the same problem in private.

What had not been built — what nobody had quite found the courage to propose — was a protocol that any model could speak and any service could expose, with nobody in particular owning the middle. There had been attempts at adjacent shapes. OpenAPI was, in principle, a candidate, but its mental model had been formed around the conventions of REST and it required more ceremony than a tool-calling agent could justify. The W3C had sketched a few standards in the general vicinity. None of them had the right scope: small enough to implement in an afternoon, open enough that no single vendor could capture it, and concrete enough that two strangers shipping against it on different continents would, mechanically, interoperate. MCP, when it arrived, was that proposal. The surprise was not that it existed. The surprise was that the company that wrote it was willing to walk away from the lock-in it could have built instead.

II · The SpecThe three primitives, expanded.

Most protocols accumulate features. MCP was designed, visibly, against that gravity. The first version shipped with three primitives — tools, resources, and prompts — and the second version has, almost defiantly, the same three. The temptation to add a fourth must be considerable; the discipline of refusing has, I suspect, done more for adoption than any single technical decision. Each primitive corresponds to a different stance the model can take toward the world, and the boundary between them is doing real work.

Tools — the verbs.

A tool, in MCP, is a function the model is allowed to invoke. Each tool declares a name, a human-readable description, and an input schema written in JSON-Schema. The schema is not decorative. It is what allows the model to construct a valid call on the first attempt without coaxing — which, when you are paying per token and waiting on a network round trip, is the difference between an agent that feels fluent and one that feels like a drunk receptionist. The model receives the list of available tools as structured context at the start of a session, and it may, at any point in the conversation, decide that it would like to call one. The server executes, returns a result, and the model integrates the result into its next turn.

What is interesting about tools is the discipline of what they are not. They are not RPCs, in the strict sense, because there is no expectation that a tool corresponds to a single backend function: a server author is free to compose, validate, retry, and reshape underneath the surface that the model sees. They are not webhooks: the model initiates, the server responds. They are not capabilities in the object-capability sense: there is no token-passing, no transitive delegation. They are, deliberately, just functions — the smallest unit of action the runtime knows how to permission and audit. Side effects belong here, and only here. If a primitive can change the world, it is a tool.

The reason this matters becomes obvious the first time you try to reason about safety. A user reviewing a Claude Desktop session knows, with no further interpretation required, that anything in the tools list could mutate state. The mental model for granting permission is the same as the mental model for granting access to a CLI utility: I am giving this agent the ability to do these specific things. If tools shared a surface with read-only data, that line would blur. By keeping side effects in their own primitive, MCP makes the question "what could this server do to me?" answerable by reading a single list. That is not a small thing.

Resources — the nouns.

A resource is something the model can read. It is identified by a URI — file:///repo/README.md, postgres://orders/2026-04-12, github://issues/open — and it returns content, optionally with a MIME type, optionally watchable for change. Resources are explicitly read-only. They cannot mutate. They can, however, be subscribed to: a client may ask the server to notify it when the resource changes, and the model can incorporate the freshest version into the next turn without anybody re-prompting. This is a more useful primitive than it sounds, because it lets the server own the question of when data is current rather than offloading that question to the prompt author.

The choice to separate resources from tools is the part of the spec I find most thoughtful. In every previous tool-calling system I have used, "read this file" and "delete this file" were both functions, distinguished only by their names and their descriptions. The framework had no structural way to tell them apart, and so the user had no structural way to grant the model "read everything but write nothing" — a permission posture that, in practice, is what people actually want. By giving resources their own primitive, MCP makes that posture native. A client can connect to a server and request access to its resources without ever loading its tools. The principle of least privilege, which used to require a custom proxy layer, now requires a checkbox.

Resources also push the burden of freshness in the right direction. When the model wants to know the current state of an issue tracker, it does not have to be coached to call a function and remember to call it again; it can subscribe to the resource and trust the server to push updates. This is the same pattern that made reactive UI frameworks pleasant: declare what you want, let the system handle change detection. It is a small thing, but a small thing applied across ten thousand servers becomes the difference between agents that get stale and agents that stay synchronized with the world they live in.

Prompts — the recipes.

The third primitive is the least appreciated. A prompt, in MCP, is a reusable template the server offers to the client. It is not a tool the model calls; it is a workflow the user invokes — usually through a slash command, a menu item, or a button in the host application. When invoked, the prompt expands into a structured message (sometimes with arguments, sometimes with embedded resource references) and is handed to the model as the starting point of a conversation. The Filesystem server, for example, might offer a /summarize-directory prompt; the GitHub server might offer /review-pr; a custom analytics server might offer /weekly-digest.

It is tempting to dismiss this as syntactic sugar over good prompts, and many early MCP clients did. The promise becomes clearer once you have lived with a few servers that ship thoughtful prompts. A well-authored prompt encodes the server author's expertise about how their own tools should be combined — which sequence of calls produces a useful result, which framing nudges the model away from common failure modes, which output format the user is likely to want. A good GitHub MCP server ships a /triage prompt that knows to fetch the open issues, group them by label, summarize the top recurring themes, and propose a triage order. The user types two words. The model does the rest. Without the prompt primitive, that expertise would live in a README that nobody reads.

Prompts also matter because they invert the discovery problem. New users of a tool-calling system typically struggle to know what the system can do: the tools list is exhaustive but inscrutable, and the documentation rarely keeps up. Prompts solve this by presenting a curated, server-authored menu of "things worth asking for." They are the table of contents of the agent's capabilities, written by the people who built them. That is a small reorientation, but it is the reorientation that makes a server feel like a product rather than a kit.

III · The WireOn stdio and HTTP+SSE.

Underneath the three primitives sits a transport layer, and the choices made there are revealing about the protocol's temperament. MCP supports exactly two transports. The first is stdio — the venerable Unix convention of speaking JSON-RPC over a child process's standard input and standard output, with the parent process spawning the child and reading its replies line by line. The second is HTTP+SSE — POST for client-to-server messages, Server-Sent Events for the server-to-client stream. That is all. There is no native WebSocket transport, no gRPC, no Protocol Buffers, no QUIC, no custom binary framing. The protocol designers looked at the menu and chose the two boring options.

The case for stdio is that it is, in practical terms, the cheapest possible IPC mechanism. There is no port to allocate, no certificate to mint, no firewall rule to add. A server is a process; you spawn it; you pipe to it; when you are done, it exits. This makes local servers — the kind that read your filesystem, talk to your local database, hold your SSH keys — trivial to deploy and trivial to sandbox. The host application controls the lifecycle, the credentials live in the host's environment, and the entire attack surface is the JSON the child writes to stdout. Compared to running a local HTTP daemon on a port and remembering to firewall it correctly, this is an order of magnitude less rope with which to hang yourself.

The case for HTTP+SSE is that it is what the rest of the web speaks. Every load balancer, every CDN, every reverse proxy, every auth gateway already knows how to handle POST and SSE. You can put an MCP server behind Cloudflare; you can host one on Vercel; you can multiplex thousands of clients onto one server with the infrastructure your ops team already operates. The absence of WebSockets is conspicuous, and the reason is mostly mundane: WebSockets are more complex to terminate at the edge, more prone to silent disconnect, and offer few real advantages for the request-shaped traffic MCP actually generates. SSE is one-way streaming, which is what server-to-client push wants to be anyway, and POST is request/response, which is what client-to-server calls want to be. The absence of gRPC has a similar shape — gRPC is a wonderful protocol for service-to-service traffic in a controlled deployment, but its binary framing, its HTTP/2 dependence, and its proto compilation step impose a tax that is wildly disproportionate to the rate of traffic an agent actually generates. MCP declines to be clever about transport precisely because cleverness about transport is where the bodies are buried.

IV · A Walk-ThroughBuilding a Notion MCP server.

The fastest way to understand what MCP feels like is to write a small server. Suppose you want Claude to be able to search your Notion workspace, fetch the contents of a page, and create a new entry in a database — three operations that previously would have required a fragile wrapper, a credentials file you would feel guilty about, and a prayer. With MCP, the same functionality is about sixty lines of Python, most of which is the Notion API client you would have written anyway. The protocol-shaped portion is small enough to fit on one screen.

You begin with a server. You give it a name. You attach the three things the client cares about: a tool that takes a query and returns search hits, a resource template that exposes any page in the workspace by its ID, and a prompt that wraps a common workflow — say, drafting a weekly review from a database of completed tasks. The framework handles the JSON-RPC envelope, the schema validation, and the lifecycle. You handle the part that is specific to Notion.

from mcp.server import Server
from mcp.server.stdio import stdio_server
from notion_client import AsyncClient

notion = AsyncClient(auth=os.environ["NOTION_TOKEN"])
server = Server("notion")

@server.tool()
async def search(query: str, page_size: int = 10) -> list[dict]:
    """Search across all pages and databases the integration can see."""
    res = await notion.search(query=query, page_size=page_size)
    return [{"id": r["id"], "title": _title_of(r)} for r in res["results"]]

@server.resource("notion://page/{page_id}")
async def read_page(page_id: str) -> str:
    """Fetch a page's blocks as markdown."""
    blocks = await notion.blocks.children.list(block_id=page_id)
    return _to_markdown(blocks["results"])

@server.prompt()
def weekly_review(database_id: str) -> str:
    return f"""Read the database at {database_id}. Group completed
    tasks by project. Write a one-paragraph weekly review in the
    voice of a thoughtful peer."""

if __name__ == "__main__":
    stdio_server(server).run()

You register this server with Claude Desktop by adding a five-line entry to a config file: a command to run it, an argument list, the environment variable that holds the token. The next time you open the application, Claude will discover the server, list its tool and prompt in the available capabilities, and let you ask, in plain English, to "search my Notion for last quarter's planning doc and draft a weekly review." The model will call search, read the page through the resource URI, invoke the prompt, and return a draft. You have not written a single line of agent code. You have not had to think about parsing the model's JSON or handling tool errors or formatting the conversation history. You have written, essentially, a Notion adapter — and because you wrote it against MCP rather than against a specific model, every MCP-speaking client now benefits. Cursor will work. The Continue plugin in VS Code will work. The custom internal agent your colleague built last month will work. The integration is, at last, the integration.

V · The FieldThe ecosystem that emerged in six months.

It is worth pausing on the shape of what was built. Within a month of the protocol's release, Anthropic itself shipped reference servers for the Filesystem, GitHub, Google Drive, Slack, Postgres, SQLite, Puppeteer, Fetch, Sentry, Memory, and Sequential Thinking — a remarkably broad set for a single team to maintain, but a deliberate one, because each was meant less as a finished product than as a worked example of what a server should look like. The Filesystem server became, almost by accident, the most-installed piece of agent infrastructure in the world, because every Claude Desktop user installed it within a day of opening the app and most of them never uninstalled it. The GitHub server followed within weeks. The pattern, by that point, had visibly taken.

The second wave came from the SaaS vendors, and it came faster than anyone expected. Linear shipped an official server before the end of the year. Notion followed in January, then Asana, then Stripe, then Sentry's first-party version replacing the reference one. Vercel's MCP server became a way to manage deployments and logs from inside any agent; Cloudflare's exposed Workers, KV, R2, and D1; Supabase, Neon, PlanetScale, and Turso each added Postgres-shaped servers tuned to their own deployment surfaces. Upstash and Redis added cache-shaped ones. Datadog, Honeycomb, and New Relic added observability servers, which turned out to be the moment when the protocol crossed from developer-tools into operations: the on-call engineer could now ask the model, in English, what had changed in the last hour. The dashboard, after twenty years of accumulation, finally had a colleague.

The third wave was the long tail, and it was a long tail in the genuine sense — thousands of small, specific, often single-purpose servers built by individuals and small teams to scratch their own itch and casually published to a registry. Context7 began as one of these and grew into something close to essential: a continuously updated index of library documentation, callable as a resource, so the model is no longer reasoning from a 2024 training cutoff when you ask it about a 2026 library. PyPI got a server. Homebrew got one. The Wolfram Alpha server made the model genuinely good at arithmetic for the first time. The Excalidraw server let the model produce diagrams. Someone wrote a server that exposes the ephemerides of every planet so an astrology app could ask Claude about transits, and someone else wrote one that exposes the entire arXiv corpus so a researcher could ask Claude about the state of a field. Neither of these was a sensible thing to build, in the older calculus where every integration was expensive. In the newer calculus, they took an afternoon, and now they exist, and a small number of people are quietly grateful for them every day.

By the spring of 2026, the public registries indexed more than ten thousand servers, and the rate of new submissions had stopped being a useful metric — much in the way that the rate of new npm packages stopped being a useful metric around 2017. The protocol had crossed the threshold at which its growth was no longer a story about itself. It had become, instead, a story about everything else: about Cursor, about Windsurf, about Zed, about Claude Desktop and the half-dozen clones that followed, about the internal agents that companies stopped writing from scratch and started assembling from off-the-shelf servers like Lego. The MCP-shaped part of any given project shrank, in proportion, until it was no longer the part you talked about.

VI · ConsequencesWhat changed when MCP shipped.

The honest answer to "what changed" is that several categories of work became cheap enough to attempt that nobody had previously bothered to attempt them. Consider the workflow of preparing for a sales call. Before MCP, this was a half-hour of clicking: open the CRM, open the email thread, open the company's website, open the last quarterly report if you could find it, open LinkedIn for the person you were meeting, open Notion for the playbook your team had written, and somewhere in there assemble a mental picture of where the conversation should go. After MCP — with HubSpot, Gmail, the Fetch server, and a Notion server all connected to the same agent — this becomes one sentence: prepare me for my 2pm with Acme. The model reads what it needs, drafts a one-page brief, and the half-hour becomes thirty seconds. None of the components are new. What is new is that they finally share a hallway.

Or consider production incident response. Before MCP, an on-call engineer at 3am had to log into PagerDuty, follow the link to the Datadog dashboard, click through to the failing service, copy the trace ID, paste it into Honeycomb, search the GitHub repo for the last deploy that touched that service, open the PR, read the diff, and then — only then — begin to reason about what to do. After MCP, with the Datadog, Honeycomb, GitHub, and PagerDuty servers connected to Claude, the same engineer can say what broke? and get a paragraph back that names the service, identifies the regression, points to the commit, and suggests a rollback. The reasoning is not new. The credit goes to the engineer, not the model. What MCP supplies is the connective tissue that was previously a tab-switching ceremony.

A third example, more quiet but more telling: long-form research. Before MCP, asking a model to "read these twenty papers and tell me what the field thinks about X" required either a custom RAG pipeline or a lot of copy-pasting. With the arXiv server, the Zotero server, and the Context7 server, the model can pull abstracts, fetch full PDFs when needed, cross-reference against current library documentation, and assemble an annotated literature review in a single session. This is not a workflow that scales — token limits still apply, the model still hallucinates if you push it past what it knows — but it is a workflow that exists now, and it did not exist a year ago. The cost of curiosity went down by an order of magnitude, and we are still, I think, underestimating what that means.

VII · Honest LimitsWhat MCP doesn't do (yet).

It would be a disservice to the protocol — and to the people considering whether to bet on it — to write only the appreciative half of the essay. MCP is, in the spring of 2026, still a young protocol with visible gaps. The most consequential of these is its stance toward state. MCP is largely stateless at the protocol level: each request stands more or less on its own, and the server is not expected to maintain a session-specific memory across calls. This is a design choice that pays for itself in simplicity, but it pushes the question of stateful workflows — long-running jobs, partial results, resumable computations — out of the protocol and into whatever the application layer is willing to build. For many use cases this is fine. For agent workflows that run for hours and need to survive a client disconnect, it is not yet a solved problem.

Closely related is the absence of native streaming for tool results. A tool, in MCP, returns its result when it finishes; there is no first-class way to stream partial output back to the model as it is produced. This means that a long-running tool — a build, a deploy, a slow query — must either block until completion or be split into a series of polling-shaped calls that pretend to be incremental. SSE handles the server-to-client direction at the transport layer, but the primitive shape of "tool returns thing" is still atomic. The community is actively working on this; the spec's evolution toward streaming responses is one of the threads worth watching closely in the next year.

The other unfinished area is authentication and multi-tenancy. The current spec assumes that a server runs in a trusted context — a user's local machine, or a small team's deployment — and that whoever connects to it is whoever it expects to be talking to. There is no formal OAuth flow baked into the protocol itself; servers that want to authenticate users do so out-of-band, which works fine for hosted SaaS like Linear or GitHub but is awkward for the kind of multi-tenant, per-user-credential server design that an enterprise eventually wants. Capability negotiation, similarly, is informal: a client introspects what the server offers, but there is no rigorous way for either side to declare what it requires. These are tractable problems. They are also not yet solved, and the working groups arguing about them in public have not yet reached the level of agreement that lets the rest of the ecosystem stop holding its breath.

VIII · A Field WalkThrough the ecosystem.

The servers that have come to feel most consequential are not, in general, the impressive ones. They are the boring ones — the ones that make a small piece of an existing workflow disappear into the model. Filesystem and GitHub are the two most-installed by a wide margin, because they convert the daily friction of "let me find that file" and "let me check that PR" into a sentence. Google Drive and Notion serve a similar purpose for teams whose knowledge lives somewhere other than a repo. None of these are novel categories of software. What is novel is that the model can now act inside them without a brittle integration sitting between.

A second cluster is more telling about where the protocol is headed. Linear, Slack, Asana, and Vercel wrap the parts of a working day that used to require context-switching between half a dozen browser tabs. Context7 exposes a continuously updated index of library documentation, so the model is no longer reasoning from a training cutoff. Stripe, Sentry, and Postgres push the protocol into operational surfaces — billing, error monitoring, production data — that used to be the territory of dashboards and on-call engineers. The shape of what one can ask the model to do has visibly widened.

Some of these servers are official, maintained by the vendor whose product they wrap. Others are community efforts, sometimes better than the official version. This is one of the more interesting consequences of MCP being a protocol rather than a marketplace: there is no central gatekeeper, no review queue, no rev-share. If you want your favorite tool to be addressable by an LLM, you write the server, you publish it, and the model can use it tomorrow. The registry is a discovery aid, not a permission system. Whether this will hold under load is one of the open questions of the next few years, but the early texture of it has been refreshingly old-internet.

IX · ClosingThe protocol that needs less ceremony.

What MCP gets right, in the end, is not technical so much as temperamental. It refuses to be impressive. It does not introduce new abstractions where existing ones — JSON-RPC, JSON-Schema, stdio, HTTP — already do the job. It does not try to be a runtime, an agent framework, a memory store, or a model itself. It is a protocol, in the older and humbler sense of the word: a small set of agreements that lets independent parties cooperate without having to know much about one another. That is the kind of thing that quietly becomes infrastructure, and infrastructure, when it works, is the thing you stop thinking about.

The conventional wisdom, for most of the last three years, has been that the path to more capable AI runs through more capable models — bigger, smarter, better post-trained, more deeply reasoned. That is true, as far as it goes, and the model labs will continue to do their work. But the second half of the sentence, the half MCP makes legible, is that capability is not only a property of the model. It is a property of the model's access — to the files, the systems, the data, the tools that constitute the world the model is being asked to act in. A small model with the right servers is, in many cases, more useful than a large model with none. The arithmetic of capability turns out to depend, at least partly, on the arithmetic of integration; and the arithmetic of integration, before MCP, was multiplication. It is not the model that needs more capability. It is the protocol that needs less ceremony. With ceremony reduced — with the integration problem made boring at last — the surprising thing is how much of the work was already, quietly, waiting to be done.

The work ahead is mostly social. The protocol exists; the SDKs exist; the registry is growing on its own. What remains is the slow accumulation of trust — that this server does what it says, that this tool will not delete your repo, that this resource is fresh — and the conventions that grow up around shared infrastructure once enough people are leaning on it. We are early in that accumulation. But the floor is poured, and we are, for the first time in this corner of the field, building on it together.
